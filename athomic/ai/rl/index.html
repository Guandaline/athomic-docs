    Reinforcement Learning (RL)

    Overview

    The Reinforcement Learning (RL) module (nala.athomic.ai.rl) provides a robust, production-ready framework for implementing adaptive decision engines. It allows the AI system to learn from interactions, optimize choices based on feedback (rewards), and dynamically adjust its behavior over time.

    Unlike traditional heuristic rules, this module leverages mathematical policies to balance Exploration (trying new actions to discover better outcomes) and Exploitation (choosing the best-known action to maximize immediate reward).

    Scope — read this first. The RLAlgorithmProtocol abstraction is RL-generic, but the algorithms shipped today are contextual / multi-armed bandits, not classical Reinforcement Learning. EpsilonGreedyBandit is a stateless multi-armed bandit; ContextualBandit is an online linear model trained via SGD on the context vector. Neither models temporal dynamics — there is no Bellman equation, discount factor γ, TD-error, state-dependent value/Q-function, episodes, or bootstrapping. In practice this module is an Adaptive Decision Engine based on contextual bandits and online incremental learning, which is the right tool for recommendation, routing, tool/agent selection, and controlled exploration. Evolving to full RL (Q-Learning → Double Q-Learning → DQN → PPO) is additive on the existing abstraction. See ADR-139.

    Key Features

    • Stateless Architecture: Ensures absolute thread safety and zero race conditions in high-concurrency asynchronous environments. The mathematical core is pure and decoupled from persistence.
    • Pluggable Algorithms: Easily switch between multi-armed bandit strategies (e.g., Epsilon-Greedy, Contextual Bandit) via configuration without altering business logic.
    • Dual-Strategy Training: Supports both Online (synchronous, real-time weight updates) and Batch (asynchronous, offline Experience Replay) learning strategies to optimize latency versus accuracy.
    • Dual-Store Memory Facade: Intelligently splits persistence—fast Key-Value stores (Redis) for mathematical weights, and Vector/Document stores (Qdrant) for rich Experience Replay traces.
    • Full Observability: Deeply instrumented with OpenTelemetry traces and Prometheus metrics tracking exploration rates, reward distributions, and operation latency.

    Architecture

    The RL module strictly adheres to the Strategy and Template Method patterns, isolating mathematics, persistence, and execution flow.

    1. Service Layer (ReinforcementLearningService): The orchestrator. It manages the lifecycle and delegates operations between the algorithm, the memory, and the trainer. It holds no mathematical state.
    2. Mathematical Layer (RLAlgorithmProtocol): Pure, stateless implementations of RL policies. They receive their state (weights) via arguments, compute the next action or the updated weights, and return them.
      • Epsilon-Greedy: Simple, effective multi-armed bandit.
      • Contextual Bandit: Uses Stochastic Gradient Descent (SGD) and state vectors to make context-aware decisions.
    3. Persistence Layer (RLMemoryService): A facade over the database manager.
      • Saves/Loads weights using the KV Store (KVStoreProtocol).
      • Stores/Samples experiences (State, Action, Reward, Next State) using the Vector Store (VectorStoreProtocol).
    4. Trainer Layer (RLTrainingStrategy): Defines when and how the algorithm updates its weights.
      • Online: Updates weights immediately during the user request (blocks until saved).
      • Batch: Saves the experience instantly (low latency) and defers weight updates to an offline background job via train_offline.

    Usage Example

    Basic Online Learning (Epsilon-Greedy)

    This example demonstrates how to use the RL service to make a decision and immediately provide feedback.

    ```python import uuid from nala.athomic.ai.rl.factory import ReinforcementLearningFactory

    async def optimize_ui_layout(): # 1. Initialize the RL Service (automatically wires DBs based on settings) rl_service = ReinforcementLearningFactory.create() await rl_service.connect()

    # The unique identifier for this specific learning context
model_id = "homepage_banner_optimization"

# 2. Get an Action (Predict)
# State vectors are optional for Epsilon-Greedy but required for Contextual Bandits
action, metadata = await rl_service.get_action(
    model_id=model_id,
    state_vector=[1.0], 
    available_actions=[1, 2, 3] # e.g., 3 different banner designs
)

print(f"Selected Banner ID: {action} (Explored: {metadata.get('exploration_triggered')})")

# ... application displays banner and waits for user interaction ...

# 3. Process Feedback (Update)
# E.g., The user clicked the banner (Reward = 1.0) or ignored it (Reward = 0.0)
user_clicked = True 
reward = 1.0 if user_clicked else 0.0

await rl_service.process_feedback(
    model_id=model_id,
    state_vector=[1.0],
    action_index=action,
    reward=reward,
    next_state_vector=[1.0]
)

    ```

    Offline Batch Training (Experience Replay)

    For high-throughput systems, updating mathematical weights on every request adds unacceptable latency. Use the Batch trainer to simply record interactions and train later.

    ```python

    Assuming settings.toml defines trainer_type = "batch"

    During the live user request:

    await rl_service.process_feedback( model_id="recommendation_engine", state_vector=[0.5, 0.8, 0.1], # User context action_index=5, # Recommended item reward=10.0, # User bought the item next_state_vector=[0.9, 0.8, 0.1] )

    Returns almost instantly. Weights are NOT updated yet.

    ... later, in a background Celery/Taskiq worker ...

    async def nightly_model_training(): await rl_service.trigger_offline_training( model_id="recommendation_engine", batch_size=1024 # Sample a random batch from the Vector Store and apply SGD ) ```

    Configuration

    RL is configured under [ai.rl] in settings.toml. It relies on the global database configurations for KV and Vector stores.

    ```toml [default.ai.rl] enabled = true

    The mathematical policy. Options: "epsilon_greedy", "contextual_bandit"

    default_algorithm = "epsilon_greedy"

    The execution strategy. Options: "online", "batch"

    trainer_type = "online"

    --- Algorithm Hyperparameters ---

    [default.ai.rl.algorithm_parameters]

    How heavily new rewards override old knowledge (0.0 to 1.0)

    learning_rate = 0.1

    The probability of choosing a random action to discover new paths (0.0 to 1.0)

    exploration_rate = 0.2 discount_factor = 0.9

    --- Infrastructure Bindings ---

    [default.ai.rl.memory]

    Must match a connection defined in [database.kvstore]

    kv_store_connection_name = "default_redis"

    Must match a connection defined in [database.vector]

    vector_store_connection_name = "default_qdrant"

    Namespaces to isolate RL data from other application caches

    weights_namespace = "nala:ai:rl:weights" experience_collection_name = "nala_rl_experiences" ```

    Observability

    The RL Service automatically exports metrics to Prometheus to monitor the agent's learning progress and system health.

    Metric Name Type Labels Description
    rl_exploration_total Counter algorithm Number of times the agent chose a random action (Exploration) vs known best (Exploitation).
    rl_reward_distribution Histogram algorithm The distribution of rewards received, useful to detect if the environment is too sparse.
    rl_operations_total Counter algorithm, operation, status Total throughput (predict, update) and success/failure rates.
    rl_operation_duration_seconds Histogram algorithm, operation, status Latency overhead of predictions and weight updates.
    ai_rl_model_decisions_total Counter model_id, outcome Decisions served per model, split by explore/exploit. Exploration ratio = explore / (explore + exploit). Emitted by the serving plane.
    ai_rl_model_reward Histogram model_id Per-model reward stream. _sum = cumulative reward (learning-progress proxy); rate(_sum)/rate(_count) = reward moving average; bucket spread ≈ variance. Emitted by the training plane.

    On cumulative regret: true regret needs the optimal arm's expected reward, which is not available at runtime, so it is intentionally not emitted as a fabricated metric. Cumulative reward (ai_rl_model_reward_sum) is the honest proxy; a real regret baseline is deferred to shadow-mode execution (backlog RL-006).

    Useful PromQL

    # Per-model exploration ratio over 5m
sum(rate(ai_rl_model_decisions_total{outcome="explore"}[5m])) by (model_id)
  / sum(rate(ai_rl_model_decisions_total[5m])) by (model_id)

# Per-model reward moving average over 5m (is the agent learning?)
sum(rate(ai_rl_model_reward_sum[5m])) by (model_id)
  / sum(rate(ai_rl_model_reward_count[5m])) by (model_id)

    nala.athomic.ai.rl.algorithms.protocol.RLAlgorithmProtocol

    Bases: Protocol

    Defines the strict mathematical interface for Reinforcement Learning algorithms.

    This protocol ensures that the learning core remains isolated from domain-specific business logic. Implementations must operate exclusively on numerical vectors and discrete indices in a completely stateless manner, receiving their execution state (weights) per invocation.

    evaluate(weights, transitions)

    Returns a read-only offline score of weights over a held-out batch of transitions — higher is better (RL-007 pre-promote eval gate).

    weights is typed as the opaque JSON blob it is (Dict[str, object], not Dict[str, Any]): the algorithm narrows the few entries it reads.

    The score is a value-calibration proxy: how well weights predict the observed rewards of the held-out transitions. Like weight_health, the algorithm owns it — it is the only component that knows how to read a prediction out of its weight layout. The computation is pure and must not mutate weights or fit on the batch (it is an evaluation, not a training step), so the same batch can score two weight sets fairly.

    validate_weights(weights)

    Validates a raw weights blob against this algorithm's structural schema and returns a normalized, JSON-safe copy (RL-008).

    Called at the (de)serialization boundary of the weight read-modify-write cycle — right after a blob is loaded from storage and right before it is persisted — so a structurally corrupt blob (wrong value types, ragged contextual vectors, …) can never be silently written, snapshotted, or fed into a learning step. Like weight_health, the algorithm owns this: it is the only component that knows which keys its weights carry and what shape they must have. Sibling keys the blob carries beyond the learnable parameters are preserved.
