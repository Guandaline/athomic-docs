    Backlog — Architecture Scorecard Refinements (v2)

    This backlog captures the architectural and methodological improvements identified after the first successful application of the Architecture Scorecard.

    The objective is not to change the core thesis of the scorecard, but to improve metric correctness, reduce false positives, and strengthen its credibility as an architectural fitness evaluation framework.

    EPIC-01 — Correct Package-Level Martin Metrics

    Motivation

    The current implementation computes Instability (I), Abstractness (A), and Distance from Main Sequence (D) using individual source files as units.

    However, Robert Martin originally defined these metrics at the package/component level.

    As a consequence, the scorecard may currently produce distorted abstraction scores for modules composed of multiple files.

    Story 1.1 — Introduce PackageSignals aggregation

    Description

    Create a package aggregation layer that groups ModuleSignals into package-level summaries.

    Metrics derived from coupling should be calculated only after aggregation.

    Deliverables

    Implement:

    @dataclass(frozen=True)
class PackageSignals:
    package: str

    fan_in: int
    fan_out: int

    num_types: int
    num_abstract_types: int

    maintainability_index: float
    complexity: float

    Story 1.2 — Move Martin metrics to package scope

    Description

    Remove file-level computation of:

    • instability
    • abstractness
    • distance

    from ModuleSignals.

    Expose them only through PackageSignals.

    Acceptance Criteria

    Distance metric values match expectations from Martin's original definitions.

    EPIC-02 — Improve Internal Import Resolution

    Motivation

    Current import detection only matches exact module names.

    Examples such as

    from nala.ai.features import engines

    or

    from nala.ai import features

    are ignored.

    This causes underestimation of coupling.

    Story 2.1 — Support prefix matching

    Resolve imports by longest matching module prefix.

    Example

    nala.ai.features.engines.runtime

    should match

    nala.ai.features.engines

    Story 2.2 — Handle relative imports

    Support

    from .engine import Runtime

    and

    from ..common import BaseClass

    using

    importlib.util.resolve_name

    or equivalent.

    EPIC-03 — Replace SRP Proxy with Responsibility Focus

    Motivation

    Public method count alone is not a reliable indicator of SRP adherence.

    Examples exist where cohesive classes expose many methods while God Objects expose very few.

    Current implementation:

    max_public_methods

    is therefore a weak heuristic.

    Story 3.1 — Introduce Responsibility Focus dimension

    Compute a composite score based on multiple signals.

    Example formula

    responsibility_focus

=

0.40 × public_methods_score

+

0.30 × fan_out_score

+

0.20 × sloc_score

+

0.10 × collaborator_count_score

    Story 3.2 — Rename SRP terminology

    Replace terminology

    srp

    with

    responsibility_focus

    throughout the scorecard.

    Acceptance Criteria

    No dimension should claim to directly measure SRP.

    EPIC-04 — Improve Hidden Coupling Detection

    Motivation

    The scorecard intentionally treats internal TYPE_CHECKING imports as architectural smells.

    The issue is not TYPE_CHECKING itself.

    The issue is using guarded imports to conceal structural dependency cycles.

    Story 4.1 — Replace banned_type_checking metric

    Current metric

    banned_type_checking

    becomes

    guarded_internal_imports

    Story 4.2 — Classify guarded imports

    Severity matrix

    Pattern Penalty
    TYPE_CHECKING importing sibling package 10
    TYPE_CHECKING importing internal component 20
    TYPE_CHECKING importing third-party dependency 0

    Story 4.3 — Report hidden cycles

    Deduction messages should explain the issue.

    Example

    Hidden coupling

3 guarded imports detected

Potential dependency cycles concealed via TYPE_CHECKING

    EPIC-05 — Confidence Index

    Motivation

    Modules evaluated using very few architectural signals may receive artificially high scores.

    The scorecard should expose how trustworthy the final score is.

    Story 5.1 — Extend ModuleScore

    Add

    confidence: float

    Story 5.2 — Confidence computation

    Possible implementation

    confidence

=

implemented_dimensions

/

expected_dimensions

    Alternative

    confidence

=

signal_coverage_weighted

    Story 5.3 — CLI support

    Display

    Module                       Score     Confidence

service                      74.3        92%

lake                         75.4        87%

    EPIC-06 — Dimension Objects

    Motivation

    The current registry of callables works well.

    However explicit dimension objects would allow richer metadata.

    Story 6.1 — Introduce ArchitectureDimension protocol

    class ArchitectureDimension(Protocol):

    name:str

    description:str

    category:str

    def score(
        self,
        signals: ModuleSignals
    ) -> float:
        ...

    Story 6.2 — Replace DIMENSION_SCORERS

    Current

    DIMENSION_SCORERS

    Becomes

    DIMENSIONS


Dimension(
    name="complexity",
    category="maintainability"
)


Dimension(
    name="hidden_coupling",
    category="coupling"
)

    EPIC-07 — Statistical Calibration

    Motivation

    Current thresholds are hand-crafted.

    Example

    CC <= 5

CC >= 30

    These values may not reflect the actual distribution of complexity inside the framework.

    Story 7.1 — Baseline distribution analysis

    Compute

    • mean
    • stddev
    • percentiles

    for

    • complexity
    • maintainability
    • fan-out
    • responsibility_focus

    Story 7.2 — Percentile scoring

    Example

    complexity_score

=

100

-

percentile(max_cc)

    Acceptance Criteria

    Scores adapt naturally to repository evolution.

    EPIC-08 — Future Research Directions

    These items are intentionally postponed until the static scorecard stabilizes.

    Mutation Testing Fitness

    Signals

    Temporal Coupling

    Detect files frequently modified together.

    Example

    Acceptance Criteria

    Expected Outcome

    After completing this backlog, the Architecture Scorecard should evolve from:

    A useful architecture scoring experiment

    into

    A statistically calibrated, role-aware architectural fitness framework capable of expressing architectural intent, detecting hidden coupling, and enforcing long-term structural quality through CI.
