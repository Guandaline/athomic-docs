    Adaptive Concurrency & Admission Control

    • TopFull: An Adaptive Top-Down Overload Control for SLO-Oriented Microservices (2024) – Proposes TopFull, a global overload controller that uses per-API rate controllers (tuned by reinforcement learning) to throttle incoming requests based on system-wide observations. It adaptively adjusts admitted rates of each service’s API during load surges to maximize “goodput” (SLO-compliant throughput). In benchmarks, TopFull significantly boosts throughput under overload (1.8–2.3× gains over prior methods) by leveraging global visibility and learned rate limits.
    • Rajomon: Decentralized and Coordinated Overload Control (NSDI 2025) – Introduces Rajomon, a fully distributed overload-control mechanism for large microservice graphs. Clients attach tokens to requests and services charge a dynamic price per API; requests lacking tokens are dropped. Tokens and prices propagate end-to-end, effectively balancing load without a central coordinator. Experiments show Rajomon protects tail latency and goodput under spikes (reducing tail latency by 78% and improving goodput by 45% vs. prior state-of-art).
    • Load Balancers Need In-Band Feedback Control (HotNets 2022) – Argues that Layer-4 load balancers should implement in-band feedback controllers to adapt request routing in real time. Instead of shipping metrics to a controller, each load balancer observes performance of the servers it fronts and uses a simple feedback loop (much like TCP congestion control) to adjust routing. This “in-band feedback” approach can dynamically throttle or re-route traffic based on local latency observations, improving tail latency without global coordination. (This work highlights a control-theoretic approach to admission control at the network edge.)

    Fair Scheduling & Resource Allocation

    • Synergy: Resource-Sensitive Scheduling (OSDI 2022) – Shows that treating the GPU as the sole bottleneck is suboptimal for multi-tenant DNN training. Synergy infers each job’s sensitivity to CPU and memory (via profiling) and then schedules jobs with GPU, CPU, and memory jointly. By allocating auxiliary resources (CPU/memory) workload-aware instead of GPU-proportionally, Synergy improves average job completion times by up to 3.4× over GPU-only scheduling. It demonstrates that many DNNs are CPU-bound during data prefetch, so skewing resource shares based on learned profiles yields large gains.
    • “Virtual Token Counter” (VTC) for LLM Serving (OSDI 2024) – Formally defines fairness in LLM inference (accounting for both input/output tokens) and proposes the VTC scheduler. VTC maintains a token counter per client and uses a fair-queuing policy: it schedules the most underserved tenant’s request next (work-conserving). VTC provably bounds service disparity and, in experiments, achieves much better multi-tenant fairness than naive FIFO or simple rate limits. It highlights that treating tokens as a continuous resource (rather than fixed quotas) can both isolate tenants and maintain high utilization.
    • Equinox: Holistic Fair Scheduling for LLMs (2025) – Identifies limitations of single-metric fairness (like VTC’s token count) and proposes a dual-counter framework. Equinox maintains a User Fairness Counter (weighted token+latency metric) and a Resource Fairness Counter (GPU utilization and throughput). A learned prediction model (Mixture of Prediction Experts) estimates future cost, enabling a scheduler that jointly optimizes tenant latency and cluster efficiency. Equinox’s implementation cuts worst-case fairness gaps by 42% and average gaps by 86% compared to VTC, achieving near-oracle fairness in multi-tenant LLM serving.
    • GPU Scheduling Survey (Chab et al., 2025) – A broad survey of GPU scheduling algorithms, covering greedy heuristics, backfilling, multi-objective optimization, and ML techniques. It notes that Dominant Resource Fairness (DRF) and its variants have been applied to GPUs to share the “dominant” resource (often GPU count) equitably among users. The survey discusses how multi-tenant DL clusters often balance fairness (e.g. Jain’s index or slowdown metrics) with utilization, and reviews approaches like LAS (fairness via time-slicing). It provides context that no single scheduling rule fits all, and highlights DRF-like policies as a standard way to allocate GPU shares fairly over time.

    AI-Oriented & GPU Scheduling

    • SAGA: Workflow-Atomic Scheduling for AI Agent Inference (2026) – Proposes scheduling at the entire program (agent) level rather than individual LLM calls. In an AI-agent pipeline, SAGA represents each agent’s inference workflow as a single unit (“agent execution graph”) to preserve caching and reuse. It also implements session-affinity batching and an “Agent Fair Share” metric to ensure fairness across agents. On GPU clusters, this holistic approach improves utilization and latency: SAGA reduces average workflow completion time by ~2.9× compared to caching-only baselines, and defines fair scheduling across multi-step workloads.
    • DriftSched: Adaptive QoS-Aware Scheduling for LLM Inference (2026) – Addresses multi-tenant LLM inference on GPUs where token-count cost estimates drift at runtime. DriftSched classifies LLM requests (short, long) and maintains a token-budget estimate per request. It adaptively updates each class’s bias factor based on observed execution (i.e. compensating for prediction drift). The scheduler uses these updated estimates to reorder the queue (e.g. SJF) and ensures each tenant meets its latency SLO. Experiments show DriftSched maintains tight latency bounds (e.g. P95 errors <10%) and improves scheduling accuracy compared to static policies.
    • Dynamic Multi-Objective GPU Scheduling (2025) – Proposes three novel schedulers (HPS, PBS, SBS) combining multiple heuristics for multi-tenant GPU clusters. For example, HPS prioritizes shorter jobs and penalizes long jobs, and also applies batching and backfilling. These dynamic schedulers significantly boost utilization and fairness: e.g. HPS achieves ~78% GPU utilization (vs ~45–67% of static policies) while reducing scheduling delays for small jobs. This work shows that mixing efficiency-oriented (packing, batching) and fairness-oriented (penalties, backfill) strategies yields far better results on mixed workloads.
    • Tian Pan’s “Why FIFO Is Wrong” (2026) – A practitioner’s analysis highlighting how naive FIFO scheduling fails for multi-tenant LLM inference (queueing large-batch prompts starves interactive queries). This analysis cites an “OSDI 2024” result (the VTC work above) and introduces Token-Weighted Fair-Share queuing, where tenants accrue “service tokens” (heavier weight for decode tokens) and the scheduler picks the tenant with the least service so far. The blog discusses “Equinox” as a dual-counter improvement. While not an academic publication, it distills the key observation that GPU memory and token-phase heterogeneity require specialized fair-queuing policies beyond standard HPC assumptions.
    • GPU Scheduling Techniques Survey (Ye et al., 2024) – An ACM Computing Surveys article that classifies deep learning training and inference schedulers. It reviews how traditional HPC schedulers (e.g. FIFO, LAS, backfill) are adapted for DL workloads, and covers specialized systems (Tiresias, Pollux, Gandiva) that assume long-running jobs or allow sharing. It also examines how new inference-specific factors (e.g. heterogeneous sequence lengths, batching) enter scheduling decisions. This survey highlights that DL workloads often optimize for both throughput and per-job latency/SLA, and summarizes challenges like unpredictable inference cost and data dependencies in training jobs.

    Learned Resource Management

    • DRL Scheduling Survey (Gu et al., 2025) – A comprehensive review of deep reinforcement learning for cloud/job scheduling. It covers value-based (Q-learning), policy-based, multi-agent, and advanced DRL methods applied to both task scheduling and resource provisioning. The survey notes that DRL can learn adaptive policies from observations and lists practical applications (e.g. autoscaling, multi-tenant task scheduling). This work points out that DRL often outperforms static heuristics in dynamic clouds, but also highlights challenges like training overhead and generalization.
    • GNN-RL HPC Scheduler (2024) – A student project proposes a GNN+RL scheduler for HPC workloads. Jobs and their dependencies are encoded as a graph; a GNN computes job embeddings, and an RL agent (e.g. PPO) learns to schedule/allocate resources based on these embeddings. This approach adapts to varying workloads by training on historical logs. In simulation on Argonne workload traces, the GNN-RL scheduler significantly improves utilization and throughput (e.g. ~2× better utilization than backfilling) while maintaining fairness. (This illustrates a trend of using learned models to optimize multi-job schedules in HPC.)
    • DeepRM & Decima (foundational RL schedulers) – Earlier works showed learning-based schedulers can match or exceed heuristics. DeepRM (Mao et al. 2016) used policy-gradient RL for online multi-resource packing; Decima (Vashishta et al. 2018) used a graph neural network to capture job dependencies in DAG workloads. Both demonstrated that a learned scheduler can adapt to workload changes and reduce average completion times compared to hand-tuned rules. (These are often cited as proof that “RL schedulers can outperform traditional ones when given accurate inputs.”).
    • Data Center Cooling (2023) – Not directly scheduling jobs, but Google deployed a model-predictive RL controller to manage data-center cooling. It optimized temperature and power use across chillers. This case study proved that learned models (trained in simulation) can safely control real infrastructure in production. It suggests that control-theoretic RL approaches are viable for resource management tasks beyond pure scheduling.

    Control Theory in Scheduling

    • AntMan (SIGMETRICS 2020) and Pollux (SIGCOMM 2020) – Both systems use feedback-control loops to adapt resource allocations for DL training jobs. AntMan monitors training throughput for each job and adjusts its share of CPUs/GPUs to meet latency objectives. Pollux similarly detects “diminishing returns” on job performance as resources change and redistributes GPUs to improve overall goodput. In effect, they implement a simple proportional feedback loop per job: if a job’s efficiency drops, slow its allocation; if it’s starved, boost it. These examples show how classic control concepts (measure error and adjust resource mix) can tune multi-job schedules at runtime.
    • HotNets: In-Band Load Balancer Control (2022) – As noted above, proposes embedding feedback control in load balancers using only local metrics. This is a control-theory idea (borrowing from TCP) applied to request admission/routing, treating the load balancer as a controller that throttles demand based on observed latency.
    • Proactive Serverless Scheduling with MPC (arXiv 2025) – Uses Model Predictive Control to mitigate cold-start latency in serverless functions. The controller forecasts future invocations and jointly optimizes container pre-warming and request dispatching to minimize tail latency under resource constraints. Experiments show up to 85% reduction in tail latency and 34% less resource use versus reactive baselines. This work exemplifies a formal control approach (solving an optimization at each interval) in a distributed admission-setting.
    • Survey on Feedback Control in GPU Scheduling (Chab et al. 2025) – As part of its survey of GPU scheduling, the paper discusses “adaptive scheduling via feedback control”. It notes that some GPU schedulers monitor key metrics (e.g. utilization or slowdown) and adjust scheduling parameters on the fly (for instance, if utilization is low the scheduler might pack more jobs, or if slowdown is high it might prioritize short jobs). This general concept—applying control loops to admission or preemption—is advocated as an alternative to static heuristics.

    Literature Gap (Unified Scheduler): We found no existing system that integrates all five aspects (adaptive concurrency control, multi-resource fairness, learned cost modeling, distributed coordination, and formal control) into one application-layer scheduler. Some works combine a few elements (e.g. TopFull merges RL with global admission control; Rajomon adds distributed coordination to rate-limiting; Synergy ties resource learning with scheduling). However, none attempt a holistic unification. In particular, we saw no published design that combines congestion feedback, DRF-like fairness, ML-based cost predictions, decentralized coordination, and control-theoretic guarantees in one end-to-end scheduler. The proposed “Athomic” concept appears to fill a gap in literature.

    Notable Surveys: Key surveys covering AI serving and resource management include: Ye et al.’s survey on Deep Learning Workload Scheduling (ACM Comput. Surveys 2024), which systematically reviews GPU cluster schedulers for training and inference; Gu et al.’s survey on DRL for Job Scheduling (2025), which overviews learning-based resource managers; and Heisler et al.’s techRxiv survey “LLM Inference Scheduling” (2025), focusing on scheduling frameworks and tradeoffs for serving large language models. These provide broad context on scheduling techniques and highlight challenges at the intersection of AI workloads and cluster resource management.

    Sources: The above summaries cite recent papers and preprints (2020–2026) on microservices overload control, multi-tenant GPU scheduling, LLM serving fairness, learned scheduling, and feedback/MPC control. These capture the state of the art and expose the lack of a comprehensive unifying scheduler.
