    Guardian v2 — deferred deep improvements

    Status: DONE · Owner: platform/quality · Created: 2026-06-30 · Closed: 2026-06-30 (all seven items shipped: #1 mutation testing, #2 executable invariants, #3 dossier auto-update, #4 regression KB, #5 committed trend, #6 live calibration,

    7 evidence-weighted verify)

    Context

    The Guardian v2 increment landed the high-ROI pieces across all four improvement fronts: hardened facts (patch-coverage, real before/after benchmarks via a base-ref worktree), a security judge (bandit) + adversarial verification of BLOCK findings, dossier staleness detection, and a verdict-history / trend reporter. This backlog tracks the heavier or more research-y pieces that were intentionally deferred so the first increment stayed shippable and tested.

    Deferred items

    1. Mutation testing (front B) — "are the tests actually effective?"

    Patch-coverage proves a line is executed; it does not prove a test would fail if the line were wrong. Add a tiered collector that runs mutmut (or cosmic-ray) on only the functions changed by the task for critical modules, and reports surviving mutants as a test-effectiveness gap. - Needs a dev dependency (mutmut); add to pyproject dev group. - Heavy → tier it: on-request or only for modules flagged critical in their dossier; cap wall-clock.

    2. Executable invariants (front C) — dossiers as checks, not prose — ✅ DONE

    Each dossier's "Behavioral invariants" were prose; now promoted to: - an explicit checklist the regression-auditor verifies item by item: invariant-checklist.sh + lib/invariants.py parse each dossier's Behavioral invariants into the invariant_checklist fact (folded by collect-metrics in every tier); the auditor returns an invariant_checks array (upheld/violated/not_applicable) and a violated Behavioral invariant is a block. Validated across all 6 dossiers (7–10 invariants each). - real Hypothesis property tests for named invariants, committed under the module's tests: tests/unit/nala/athomic/ai/rl/algorithms/test_property_statelessness.py encodes the ai/rl Statelessness invariant for both shipped bandits — proven to FAIL on a reintroduced in-place-mutation regression and PASS on the fixed code. (Further invariants — RRF formula, plan-hash determinism, offset-commit ordering — can be added the same way per module.)

    3. Auto-update "Where we stand" (front C) — ✅ DONE

    A dedicated post-PASS step keeps the bar tracking reality without a full /module-dossier re-run: - a new guardian-dossier-updater agent makes a narrow, evidence-bound edit to the touched module's Where we stand table (only rows the diff moved; never re-researches or inflates — proven on a no-op comment to return rows_updated: []); - dossier-stamp.sh then re-stamps module_tree_sha and bumps _Last assessed: to today (via the unit-tested lib/dossier.py::bump_assessed_date), so dossier-staleness goes green for the just-reviewed change. The Guardian skill runs this only on PASS (the bar must not advance on an unproven change) and skips it when the updater reports no movement.

    4. Regression knowledge base (front D) — ✅ DONE

    A committed KB (devtools/guardian/kb/regressions.jsonl) of confirmed-and-fixed regressions, each with a signature (added-line tokens), fix, and confirmed provenance. On every run kb-lookup (folded by collect-metrics, all tiers) emits the regression_kb fact for the touched modules and flags when the diff's added lines match a known signature (a recurrence); the regression-auditor treats a signature_matched entry as a priority checklist item and blocks unless the new code is proven safe. kb-record.sh appends new confirmed blockers (the Guardian skill prompts for this after a survived-and-fixed BLOCK). Seeded from the #6 live-calibration blockers (RL in-place-mutation + weakened-statelessness- test). Validated: 11 unit tests + a live run that flags the reintroduced statelessness pattern as recurrences: 1.

    5. Shared/committed trend (front D) — ✅ DONE

    An opt-in committed aggregate now complements the per-machine history: - devtools/guardian/trend/snapshots.jsonl (tracked) records repo-wide health snapshots {date, commit, archscore_mean, coverage_pct?, guardian_pass_rate_pct?}. - guardian-snapshot.sh collects one (live archscore mean via --json, optional coverage, local PASS-rate); lib/trend.py (unit-tested) computes the first→last→delta trajectory per metric. - guardian-trend.sh now renders the committed trajectory alongside the local verdict history (works even with no local history). - .github/workflows/guardian-trend.yml runs it weekly (opt-in: commits only when the GUARDIAN_TREND_SNAPSHOTS repo variable is true). Validated: 8 unit tests + a live snapshot (archscore mean 89.5) rendered by guardian-trend.sh.

    6. End-to-end calibration (front A) — ✅ DONE (live run completed 2026-06-30)

    A dry-run calibration was performed (2026-06-30): the judge prompts + adversarial-verify mechanism were run via general-purpose agents (the custom .claude/agents only register at session start) over controlled diffs on ai/rag/query_expansion/hyde_generator.py, with the deterministic diff-context collector feeding .guardian/diff.patch.

    Results:

    Test Setup Outcome
    Recall diff with 8 planted defects (3 regressions, 2 security, 1 incomplete-goal, 1 missing-test, 1 untyped) 8/8 detected, all 4 judges BLOCK, 0 false negatives, +1 unplanted real find (a possible TypeError from a new kwarg)
    Precision genuinely behavior-preserving refactor with a scary-looking restructure 0 false-positive BLOCKs — Security/Regression/Goal PASS, Test-integrity WARN (pre-existing coverage gap, correctly not a BLOCK)
    Verify — false block a fabricated BLOCK ("removed empty-content fallback") fed to 3 skeptics 3/3 refuted → the false block is correctly killed
    Verify — true block a real regression (removed the enabled short-circuit) fed to 3 skeptics 2/3 NOT refuted → block survives (majority rule, conservative). Notable: the 1 dissenting skeptic was more correct — it traced the sole caller (RAGService._expand_query_hyde) which already gates on enabled, so the removed guard was redundant defense-in-depth → the true verdict is arguably WARN, not a hard BLOCK

    Validated: high recall, high precision, the verify stage kills false blocks and (conservatively) keeps true ones; the dossier-as-bar was actively used (a judge cited the ai/rag fail-open invariant); findings are file:line-located.

    Live run — DONE (2026-06-30). The full flow (deterministic facts → real guardian-* judge panel → adversarial verify → synthesizer) was run in a fresh session over two real ai/rl diffs: - BLOCK scenario (disguised statelessness regression + deleted guard): 6/6 judges BLOCK, both adversarial skeptics failed to refute, synthesizer BLOCK — recall + true-block-survives-verify confirmed live. - PASS scenario (the benign twin: a genuine dot-product dedup + a guarding test): 6/6 PASS, no false-block — precision confirmed live.

    The runs surfaced and fixed three real collector issues (benchmark-delta PYTHONPATH pinning, test_func_delta malformed git diff, coverage-patch whole-suite fallback + passed/exit coupling) and codified baseline-red-gate + worktree guidance in the skill. Full write-up: devtools/guardian/calibration/2026-06-30-live-e2e.md.

    7. Evidence-weighted adversarial verification (front B) — surfaced by #6

    The true-block calibration exposed a limitation: the verify stage counts votes (survives unless a majority refute), but skeptics differ in evidence depth. In the run, two skeptics confirmed "the line was deleted" while one traced the actual caller and found the claim's premise unreachable — a stronger argument that lost on a head count. Improve the aggregation to weight a skeptic's verdict by the evidence it cites (e.g. a caller/usage trace or a reproduction beats a restatement of the diff), or require the surviving finding to also pass a "is the premise reachable?" check. This makes both false-block suppression and true-block survival sharper.

    Acceptance

    Each item is independently shippable. #6 (live calibration) is ✅ done and #7 (evidence-weighted verify) is ✅ merged. Remaining order: #1 (mutation) and #2 (executable invariants) for the deepest correctness signal, then #3 (auto-update dossier), #4 (regression KB), #5 (committed trend).
