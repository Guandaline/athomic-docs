    Backlog: Adaptive Concurrency Controller Refinement (v2) ✅

    Status: ✅ DONE (verified 2026-06-20). All 8 features are implemented in src/nala/athomic/resilience/adaptive_concurrency/ and exceeded by additional capabilities (pluggable algorithms AIMD/Gradient2/SLO, circuit-breaker & throughput feedback, request deadlines, distributed fleet budget, fleet cooperative damping, active baseline probing, graceful draining).

    This backlog outlines the critical architectural enhancements required to elevate the AdaptiveConcurrencyController to an enterprise, production-grade component, incorporating advanced feedback regarding high-scale performance bottlenecks, telemetry correctness, and intelligent scheduling.

    [EPIC-01] Queue Lifecycle Management & Performance Optimization

    Feature 1: Queue Timeout Implementation (Queue TTL)

    User Story: As the system gateway, I want incoming requests waiting in the concurrency queue to time out if they exceed a specific threshold, so that the application does not waste computing resources processing "ghost tasks" after the client has already disconnected.

    Technical Specifications: - Introduce queue_timeout to AdaptiveConcurrencySettings. - Wrap the self._condition.wait_for call inside _acquire with an asyncio.wait_for. - Catch asyncio.TimeoutError to safely decrement _waiting_tasks and update metrics before propagating a clean, custom QueueTimeoutError.

    Acceptance Criteria: - Tasks waiting longer than queue_timeout must be rejected immediately. - Metrics for rejected tasks must increment on timeout events. - No memory or lock leaks must occur on a timeout event.

    Estimation: - Priority: Critical - Effort: Small (1-2 days)

    Feature 2: Mitigate Thundering Herd Bottleneck (notify_all Optimization)

    User Story: As an architectural component running under high concurrency (1000+ waiting tasks), I want the resource release mechanism to wake up waiting tasks efficiently without causing massive context-switching overhead and lock contention.

    Technical Specifications: - Analyze replacing the global self._condition.notify_all() inside _release() and _evaluate_and_adjust_limits(). - Evaluate a targeted notification mechanism (e.g., waking up only a specific number of tasks via custom step-lock signaling or managing a pool of discrete task futures/events) or transitioning the inner coordination to an optimized semaphoric pattern that supports dynamic capacity updates. - Benchmark context-switching rates using standard python load simulators before and after the optimization.

    Acceptance Criteria: - Under a load of 1000+ queued tasks, releasing a single slot must not trigger an immediate wakeup of all 1000 waiting coroutines if only one can proceed. - Eliminate CPU spikes solely generated by scheduler-driven lock contention during heavy queue evacuation.

    Estimation: - Priority: High - Effort: Large (4-5 days)

    [EPIC-02] Predictive Telemetry & Signal Accuracy

    Feature 3: OpenTelemetry Tracing Integration

    User Story: As a Platform Engineer, I want the time spent by a request inside the concurrency queue to be recorded as a separate span in our distributed tracing system, so that we can accurately diagnose latency bottlenecks.

    Technical Specifications: - Integrate opentelemetry.trace. - Create a specific span named adaptive_concurrency.queue_wait around the _acquire blocking block. - Add attributes to the span: current_limit, active_tasks, waiting_tasks, and task_cost.

    Acceptance Criteria: - Every resource request must generate a visible trace span covering its queue duration. - System metrics must correlate exactly with trace attributes during high-load tests.

    Estimation: - Priority: High - Effort: Medium (2-3 days)

    Feature 4: Memory Double-Counting Mitigation

    User Story: As an admission controller, I want the system to cleanly separate real OS-allocated memory from committed virtual memory reservations, so that the AIMD algorithm doesn't trigger false-positive panic backoffs under stable workloads.

    Technical Specifications: - Refine projected_ram_percent calculations within _evaluate_and_adjust_limits. - Introduce a mechanism to handle the temporal overlap where a task has already allocated its memory on the OS level (psutil.virtual_memory().used), but its corresponding _active_memory_bytes reservation is still fully active in the controller tracking pool. - Consider utilizing a decaying reservation window or an explicit tracking state (reserved_not_allocated vs allocated) to prevent artificial limit drops.

    Acceptance Criteria: - Stable workloads must not trigger continuous AIMD limit reductions due to over-pessimistic memory projections. - System memory metrics must remain safe under burst conditions without inducing a cascading limit degradation loop.

    Estimation: - Priority: High - Effort: Medium (3-4 days)

    Feature 5: Weighted Pressure Scoring System (pressure_score)

    User Story: As a resilience system, I want hardware signals to be processed according to their actual critical risk (e.g., RAM/VRAM saturation vs. high CPU usage during preprocessing), avoiding aggressive backoffs on non-fatal events.

    Technical Specifications: - Replace the primitive boolean OR combination of resource pressure indicators (cpu_pressure or ram_pressure or vram_pressure). - Implement a composite pressure_score algorithm where individual signals carry explicit operational weights: - VRAM Pressure: Critical (high weight / instant trigger). - RAM Pressure: Critical (high weight / instant trigger). - CPU Pressure: Moderate (low weight / requires sustained duration or acts as a multiplier). - Ensure transient CPU bursts (e.g., serialization, data compression) do not force an immediate 50% concurrency drop if memory remains completely safe.

    Acceptance Criteria: - CPU usage hitting 95% while RAM and VRAM are below thresholds must trigger a localized, graceful degradation or throttled reduction, not a catastrophic multiplicative decrease. - OOM risks (RAM/VRAM) must retain immediate, aggressive backoff protection.

    Estimation: - Priority: High - Effort: Medium (3-5 days)

    [EPIC-03] Advanced Traffic Shaping & Fairness

    Feature 6: Quality of Service (QoS) & Task Prioritization

    User Story: As a system operator, I want critical system operations (like health checks or high-priority API calls) to bypass or jump ahead of low-priority background tasks in the concurrency queue, preventing thread/slot starvation.

    Technical Specifications: - Update request_resources to accept an optional priority: int = 100 parameter (lower values indicate higher priority). - Replace the primitive implicit FIFO synchronization mechanism with a priority-aware strategy (e.g., using an internal sorted collection or asyncio.PriorityQueue semantics mapped to condition notifications). - Ensure that when _release notifies waiting tasks, the highest priority tasks evaluate their constraints first.

    Acceptance Criteria: - High-priority requests must be served before lower-priority requests currently waiting in the queue, provided resource constraints are met. - Ensure strict mitigation against starvation of low-priority tasks under continuous load (e.g., aging mechanism).

    Estimation: - Priority: High - Effort: Large (5-7 days)

    Feature 7: Task Fairness & Heavy Task Anti-Starvation

    User Story: As a large-workload client (e.g., LLM inference with task_cost=50), I want a mechanism that prevents me from being indefinitely starved by a continuous influx of small tasks (task_cost=1), even when the concurrency budget is tight.

    Technical Specifications: - Implement an inner slot-reservation or task-aging mechanism within the coordination logic. - If a heavy task is at the front of the priority/historical queue, prevent smaller incoming tasks from continuously consuming newly liberated units if doing so would repeatedly push the heavy task's execution window further back.

    Acceptance Criteria: - Heavy tasks must not remain starved indefinitely under continuous light-load saturation. - Average queue wait times must display a bounded maximum across all task_cost variations.

    Estimation: - Priority: Medium - Effort: Large (4-6 days)

    [EPIC-04] Intelligent Automation & Self-Adaptation

    Feature 8: Adaptive Cost Learning Engine

    User Story: As an application developer, I want the controller to automatically learn and adjust the resource cost of different tasks based on runtime observation, removing the need to manually hardcode brittle task_cost and required_bytes parameters.

    Technical Specifications: - Introduce an internal tracking map keyed by task/endpoint signature identifiers. - Measure actual hardware differentials (RAM/VRAM delta, processing time) upon task completion. - Implement an Exponentially Weighted Moving Average (EWMA) algorithm to calculate and update the historical cost and memory footprint profile dynamically. - Use these learned profiles as the default fallback when tasks invoke request_resources without explicit manual limits.

    Acceptance Criteria: - The controller must autonomously adjust execution limits when a specific model route begins consuming more memory due to dynamic input variations. - System metrics must publish the dynamically learned costs for full operational visibility.

    Estimation: - Priority: Low / Long-Term Enhancement - Effort: Very Large (7-10 days)
