    🧩 Engineering Task: Saga hardening — terminal failure path + framework-level step idempotency

    Status: Done Priority: High Raised by: Nala Ori tenant-lifecycle audit — resilience-robustness gap surfaced while validating that athomic covers modules/tenancy.md §2 (tenant lifecycle sagas) Relates to: resilience/sagas/ (executors, reaper) · reuses the existing integration/messaging/retry/ DLQ

    🎯 Context

    athomic's saga stack is otherwise complete — OrchestrationSagaExecutor + ChoreographySagaExecutor with compensation, SagaBuilder, a KV SagaStateRepository, and a SagaReaperService that revives stalled sagas. It is the right substrate for Nala Ori's tenant lifecycle (provisioning → active → suspended → reactivated → offboarding → exported → erased), where each transition is a multi-step saga that must roll back on failure.

    An audit for that use case found two real robustness gaps — both matter most precisely for a long-lived, reaper-revived state machine like tenant lifecycle, where "erased", "offboarding" and "provisioning" are operations you cannot afford to leave in a silently-wrong or infinitely-looping state.

    Gap 1 — No terminal failure path in the reaper (compensation can loop forever)

    resilience/sagas/reaper.py (SagaReaperService._trigger_compensation) calls executor._compensate() but does not handle the case where compensation itself fails. When a compensation step raises, the exception is only logger.exception(...)-logged; there is:

    • no max-reap-attempts cap — SagaReaperSettings has no max_reap_attempts/compensation_failure_strategy field;
    • no terminal state — the saga stays in a status the reaper keeps re-selecting, so it re-attempts the same failing compensation on every poll cycle, forever;
    • no escalation — no event, metric, alert, or routing to the existing DLQ when a saga is genuinely unrecoverable and needs manual intervention.

    For a tenant offboarding/erased saga whose compensation is broken, that means an invisible infinite loop instead of a clear "this tenant transition needs a human".

    Gap 2 — Step idempotency is not enforced by the framework (PARTIAL)

    athomic has a first-class idempotency capability (resilience/idempotency/handler.py: IdempotencyHandler.run_single_flight + batch acquire/confirm), but the saga executors do not use it. In executors/orchestration_executor.py, action and compensation are invoked directly (action(state.model_dump()) / compensation(...)) with no idempotency guard. So when the reaper re-runs a step (or a compensation) that had partially applied before the crash, correctness depends entirely on every handler author hand-rolling idempotency. For lifecycle sagas that provision external resources or mutate billing/records, a non-idempotent re-run is a double-apply bug.

    Per Nala Ori's boundary rule ("gaps go upstream"), hardening the generic saga engine belongs in athomic, not worked around per consumer.

    ✅ Deliverables

    A. Terminal failure handling in the reaper

    • [ ] Track reap/compensation attempts on SagaState (e.g. reap_attempts, last_reap_error), incremented each time the reaper drives compensation.
    • [ ] Wrap _trigger_compensation so a failing compensation is caught, the attempt counter advanced, and the error recorded on state — never a bare propagate-and-log that leaves the saga eligible for the next poll unchanged.
    • [ ] Add a terminal state (e.g. SagaStatus.COMPENSATION_FAILED / FAILED_TERMINAL) that the reaper's selection query excludes, so a saga that has exhausted max_reap_attempts stops being retried and is parked for human intervention.
    • [ ] Escalate on terminal: emit a structured "saga permanently failed" event and route it to the existing DLQ (integration/messaging/retry/ — DLQManager/DLQHandler; do not build a new DLQ) plus a metric/log so ops sees it. Reuse the existing DLQ strategies (logging_only, republish_to_parking_lot).

    B. Settings for the terminal policy

    • [ ] Extend SagaReaperSettings (config/schemas/resilience/sagas_settings.py) with max_reap_attempts (default sane, e.g. 5) and a compensation_failure_strategy selecting terminal behavior (park/DLQ/alert). Defaults must fail safe — bounded retries then terminal, never unbounded.

    C. Framework-level, opt-in step idempotency

    • [ ] Let a saga step opt into automatic idempotency: when enabled (per-step or saga-wide flag on SagaStep/SagaDefinition), the executor wraps action and compensation with the existing IdempotencyHandler, keyed by a stable saga_id:step_name[:compensation] idempotency key, so a reaper re-run of a partially-applied step is a no-op replay, not a double-apply.
    • [ ] Apply symmetrically to compensation execution (compensations are re-run by the reaper too and are just as prone to double-apply).
    • [ ] Keep it opt-in and back-compat: existing sagas that don't set the flag behave exactly as today; document that enabling it requires a KV store (same dependency contract as IdempotencyHandler).

    D. Observability & tests

    • [ ] Metrics: reap-attempt counter, saga_terminal_failed_total, and idempotency hit/miss on wrapped steps; structured logs on terminal escalation.
    • [ ] Tests:
      • a saga whose compensation always fails reaches the terminal state after exactly max_reap_attempts and is no longer re-selected by the reaper;
      • the terminal event is routed to the DLQ / emits the metric;
      • a step wrapped with idempotency, re-run by the reaper after a partial apply, executes once (replay is a no-op) — for both action and compensation;
      • back-compat: a saga without the idempotency flag runs unchanged.

    ⚙ Technical Notes

    • Reuse, don't rebuild. DLQ (integration/messaging/retry/) and idempotency (resilience/idempotency/) already exist and are production-grade — this task wires the saga engine into them, it does not reimplement either.
    • Terminal ≠ FAILED. Today's FAILED is still reaper-eligible; the new terminal state is the one that means "give up retrying, escalate". Keep them distinct so transient failures still get their bounded retries.
    • Idempotency key stability is the crux. saga_id:step must be stable across reaper re-runs (derive from persisted SagaState, not from anything regenerated per attempt), or the guard is useless.
    • Reference consumer: nala-ori-api tenant-lifecycle sagas — offboarding/erase transitions are exactly the ones where an infinite compensation loop or a double-applied step is unacceptable; validate the terminal path and step idempotency against them.

    🧪 Acceptance Criteria

    • [ ] A saga whose compensation keeps failing is retried a bounded number of times (max_reap_attempts), then moves to a terminal state the reaper no longer re-selects — no infinite loop.
    • [ ] Reaching terminal escalates: a structured event routed to the existing DLQ + a metric/log, so unrecoverable sagas are visible for manual handling.
    • [ ] Saga steps (and compensations) can opt into framework idempotency keyed by saga_id:step, making reaper re-runs safe replays; sagas that don't opt in are byte-for-byte unchanged.
    • [ ] SagaReaperSettings exposes max_reap_attempts + failure strategy with fail-safe defaults; new metrics/logs cover reap attempts, terminal failures, and idempotency hits.

    🔗 Relationship to existing backlog / capabilities

    • DLQ already exists (integration/messaging/retry/: DLQHandler, DLQManager, DLQSettings) — this task routes to it, adds no new DLQ.
    • Outbox already exists (integration/outbox/, database/outbox/) — relevant to choreography sagas' atomic publish, out of scope here.
    • Idempotency engine already exists (resilience/idempotency/) — this task wires it into the saga executors; it does not build a new one.
    • Companion audit finding hedging (a genuinely absent resilience pattern) is tracked separately in TASK-resilience-hedging.md — orthogonal to sagas.
