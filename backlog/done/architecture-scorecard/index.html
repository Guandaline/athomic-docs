    Architecture Scorecard — Objective, Scenario-Weighted Module Scoring

    Status: DELIVERED (commits 1–4) · Owner: platform/quality · Created: 2026-06-20

    Shipped 2026-06-20 in scripts/archscore/ (+ make archscore / archscore-save / archscore-check, committed baseline at scripts/archscore/baseline.json, report at docs/architecture/scorecard.md). Dimensions implemented: complexity, maintainability, abstraction (Martin distance), srp, purity. Roles: hot_path, infra, domain, adapter (added during rollout — concrete providers must not be scored like ports; the scenario-weighting thesis in action).

    First baseline findings (ai/features, mean 83.6): the scorer objectively surfaced (a) service ≈ 52 — a God facade (srp), corroborating the Control/Serving plane split in sota-roadmap.md; (b) store.providers.lake low maintainability (duckdb/pyarrow branching). These are the next objective refactor targets.

    2026-06-22 — added factory_pattern dimension (6th dimension). Triggered by a project-wide Factory audit (122 *Factory classes across src/nala/) that found 48 "trivial/off-pattern" factories — no Registry collaborator, internal get_settings() Service Locator instead of constructor injection, or a single-method factory with no provider-selection logic at all. The scorer now detects these per file (is_factory, factory_has_registry, factory_calls_get_settings, factory_trivial in ModuleSignals) and scores them via _score_factory_pattern in scoring.py — 100 (neutral) for non-factory files, with -40/-30/-30 penalties for each deviation on factory files. Weighted into every profile in archscore.toml (heaviest in infra, lightest in decorator, since factories cluster in infra/control-plane code). Baseline (ai/features, 215 modules) regenerated: mean 85.0 → 84.8 — most of that module's factories already use Registry, so the new dimension is mostly neutral here; the bulk of the off-pattern factories live outside ai/features and are tracked in factories-pattern-standardization.md.

    1. Goal

    Evaluate the architecture of each module objectively and clearly, producing a per-module score against an ideal profile that depends on the module's role (infra vs domain vs hot-path vs adapter). Today architecture quality is judged by reading code + scattered tools; we want a single, reproducible number per module with an explainable breakdown, trackable over time and gate-able in CI.

    2. Does something like this already exist? (prior art)

    Yes, in pieces — and we already use several. None gives a scenario-weighted composite per module, which is the gap.

    Tool / concept What it gives In repo?
    Architecture Fitness Functions (Ford/Parsons/Kua, Building Evolutionary Architectures) The governing idea: executable, objective checks of architectural characteristics. Our import-linter contracts already ARE fitness functions. ✅ concept partially applied
    import-linter Layering / dependency-direction contracts (5 today) ✅ make solid
    radon Maintainability Index (MI), cyclomatic complexity (CC), raw metrics ✅ make radon*
    mypy strict · pylint 10/10 · pyright · ruff · bandit Type/lint/security signal ✅
    Robert Martin package metrics — Instability (I), Abstractness (A), Distance from main sequence (D) Coupling/abstraction balance per package ❌
    LCOM / fan-in / fan-out Cohesion + coupling per class/module ❌
    SonarQube maintainability rating (A–E), tech-debt ratio Composite maintainability, but generic (not scenario-aware), heavy/external ❌
    CodeScene Behavioral "Code Health", hotspots from git history ❌ external
    ArchUnit (Java) / pytestarch Dependency rule tests partial (import-linter)

    The gap: every tool above is either single-signal (radon=complexity, import-linter=layering) or a generic composite (Sonar) that treats all code the same. None encodes that the ideal architecture differs by module role — a hot-path engine should be scored hard on complexity + allocation discipline, an infra module hard on extensibility (OCP/registry/protocol ratio), a domain module hard on SOLID/protocol purity + zero infra coupling. That scenario-weighted composite is what we build.

    3. Design — ArchitectureScorecard

    A tool (CLI + library, new top-level tools/archscore/ or scripts/) that:

    1. Collects signals per module (reusing existing tooling, not reinventing):
    2. Layering violations → parse import-linter output.
    3. Complexity (CC) + Maintainability Index → radon JSON.
    4. Type strictness → mypy/pyright clean (binary per module).
    5. Lint → pylint score.

    6. New AST-derived metrics (small ast-walk collector):

      • protocol-vs-concrete ratio (Protocol/ABC count vs concrete deps in signatures) — DIP signal,
      • public method count per class — God-class / SRP signal,
      • fan-in / fan-out (import graph) → Instability I = Ce/(Ca+Ce),
      • abstractness A = abstract types / total types; distance D = |A + I − 1|,
      • Any / cast / TYPE_CHECKING occurrences (banned-pattern signal),
      • configurability signal: Strategy+Registry / Pydantic-settings presence vs hardcoded branches (encodes [[feedback-configurability-principle]]).

    7. Scores each dimension 0–100, then combines via a scenario weight profile:

    # archscore.toml — weight profiles per module role
[profile.hot_path]        # e.g. engines/, instruction/
complexity = 0.30; allocation_discipline = 0.25; layering = 0.20
solid = 0.15; configurability = 0.10

[profile.infra]           # e.g. athomic/* factories, registries
extensibility = 0.30; solid = 0.25; layering = 0.20
type_safety = 0.15; complexity = 0.10

[profile.domain]          # e.g. ai/features transformations, training
solid = 0.30; protocol_purity = 0.25; infra_coupling = 0.20   # lower = better
type_safety = 0.15; configurability = 0.10

[module."nala.athomic.ai.features.engines"]
profile = "hot_path"

    1. Outputs: per-module score (0–100) + dimension breakdown + the top 3 deductions ("why not higher"), as JSON + a human table. Optional Markdown report committed under docs/architecture/scorecard.md.

    2. Trend + gate: persist scores (like the benchmark .tests_saves/); a CI mode fails if a module's score regresses beyond a threshold — an architecture fitness function for the composite itself.

    4. Why scenario-weighted (the original part)

    A single global "maintainability rating" punishes a hot-path engine for low abstraction (it should be concrete and fast) and rewards an infra module for simplicity (it should be extensible, which adds indirection). Weighting by declared module role makes the score reflect the intended architecture, so a high score means "ideal for what this module is", not "uniform blandness". This is the publishable/original framing — scenario-parameterized architecture fitness scoring.

    5. Sequenced epics

    1. Signal collectors — wrap radon --json + import-linter output + the small AST metric walker into a typed ModuleSignals dataclass. No scoring yet.
    2. Scoring + profiles — dimension scorers (0–100) + archscore.toml weight profiles + composite. make archscore prints the table.
    3. Report + trend + CI gate — Markdown report, score persistence, regression gate (make archscore-check).
    4. Apply to ai/features — assign profiles to its sub-modules, record a baseline, and use the breakdown to drive the next refactors objectively.

    6. Notes / open questions

    • Keep collectors pluggable (Strategy+Registry) so new signals/metrics drop in without touching the scorer — same principle the framework follows.
    • Weights live in committed config (archscore.toml), reviewed like code — the "ideal per scenario" is itself an explicit, versioned artifact.
    • Reuse, don't reinvent: radon/import-linter/mypy already produce most signals; the new code is the AST metrics + the scenario-weighted composite + reporting.
    • Consider whether git-history hotspot weighting (CodeScene-style) is a later signal once the static composite is trusted.
