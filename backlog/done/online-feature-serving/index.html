    Online Feature Serving — low-latency read path + freshness

    Status: DELIVERED (Tier 1) · Owner: features platform · Created: 2026-06-21

    Shipped 2026-06-21: enriched online envelope (store/online_envelope.py), KVOnlineFeatureReader (online_reader.py), FeatureServingPlane.get_online_features with freshness + plan_hash attestation, OnlineServingSettings (configurable miss + staleness policies), and feature_online_read_total / feature_online_age_seconds metrics. Inject a reader via FeaturePlatformFactory.create(..., online_reader=...). Closes the SOTA-roadmap online-read + freshness gaps (Tier 1).

    1. Goal

    Close the biggest parity gap in sota-roadmap.md §2: serving reads precomputed features from the online store instead of recomputing them. Today service.serving.encode(...) always computes from raw state; there is an online materialization write path (KV + TTL) but no read path. Add get_online_features(vector_id, entity_ids) that returns materialized vectors with freshness + plan_hash attestation.

    2. What exists vs what this adds

    Piece Status
    Online write: OnlineMaterializationSink → KV {prefix}:{vid}:{eid} = JSON(vector), TTL ✅ but stores bare vector only (drops the plan_hash it receives; no timestamp)
    KV read (get, exists) ✅ KVStoreProtocol
    Live serving plan_hash per vector (catalog) ✅ CatalogPlanHashResolver
    Online read path + freshness + attestation ❌ this work

    3. The differentiator (online analog of skew-free training)

    The training generator attests offline rows by plan_hash. The same idea applies online: a served precomputed vector carries the plan_hash it was materialized under; the reader can attest it against the live serving plan_hash and flag plan_mismatch (served from a stale materialization). Combined with freshness, every online read is self-describing: fresh & attested, stale, plan-drifted, or missing — no silent serving of outdated/ skewed precomputed features.

    4. Envelope change (decision)

    The online write must persist enough to attest + age. Enrich the stored value from a bare list to a small envelope:

    { "values": [..], "plan_hash": "abc123", "materialized_at": "2026-06-21T..Z" }

    OnlineMaterializationSink.materialize_batch already receives plan_hash (today discarded) — store it + a UTC timestamp. Backward read compat: a bare-list value (old format) is read as values with plan_hash=None, materialized_at=None (→ unknown freshness/attestation, never a crash).

    5. Components (SOLID)

    5.1 Port — OnlineFeatureReaderProtocol

    New serving/online_reader.py (or store/):

    class OnlineFeatureReaderProtocol(Protocol):
    async def get_online(self, vector_id: str, entity_ids: list[str]
                         ) -> dict[str, Optional[OnlineFeatureRecord]]: ...

    KVOnlineFeatureReader implements it over KVStoreProtocol (gathered get per key; get_many if the backend offers it). Parses the envelope (new + legacy).

    5.2 Result types

    @dataclass(frozen=True)
class OnlineFeatureRecord:
    entity_id: str
    values: Optional[list[float]]
    plan_hash: Optional[str]
    age_seconds: Optional[float]
    status: str   # "fresh" | "stale" | "plan_mismatch" | "missing"

@dataclass(frozen=True)
class OnlineFeatureResponse:
    records: list[OnlineFeatureRecord]   # entity-aligned
    fresh: int; stale: int; missing: int; plan_mismatch: int

    5.3 Serving-plane method

    FeatureServingPlane.get_online_features(vector_id, entity_ids): 1. reader fetches records; 2. per entity: missing → missing; else compute age = now − materialized_at; 3. attest plan_hash vs CatalogPlanHashResolver.current_plan_hash(vid) → mismatch → plan_mismatch; 4. else age > freshness_sla_seconds → stale else fresh; 5. emit feature_online_read_total{vector_id,status} + feature_online_age_seconds histogram.

    5.4 Settings (configurable, per the configurability principle)

    OnlineServingSettings:
  freshness_sla_seconds: int = 300   # age beyond which a present value is "stale"
  attest_plan_hash: bool = True      # flag plan_mismatch (advisory)

    6. Honest scope / non-goals

    • No read-through compute on miss. Online read has only entity_ids, not raw state, so it cannot recompute a miss. Misses are reported; the caller decides (fall back to encode with raw state, or skip). Documented, not hidden.
    • Advisory, not gating. stale/plan_mismatch are reported + metered; the reader never blocks serving. (A future policy could reject stale reads — seam via settings, behavior not built.)
    • Multi-pod consistency unchanged (KV is already shared).

    7. Failure modes

    Risk Guard
    Legacy bare-list values Parsed as values + plan_hash=None → unknown/skip attest; never crash
    KV down reader raises → get_online_features surfaces it (read path, caller handles); never silently empty
    Corrupt JSON per-key parse guarded → that entity = missing + warn
    Clock skew on materialized_at age clamped at ≥ 0

    8. Commit shape

    1. Enrich OnlineMaterializationSink envelope (values+plan_hash+materialized_at)
    2. legacy-read compat + tests.
    3. OnlineFeatureReaderProtocol + KVOnlineFeatureReader + result types + tests.
    4. OnlineServingSettings + FeatureServingPlane.get_online_features (freshness + attestation + metrics) + factory wiring + tests.
    5. Docs + scorecard baseline refresh.

    9. Open decisions (align before commit 1)

    • Envelope change OK? (enables attestation + freshness; legacy values still read)
    • Miss behavior: report-only (recommended) vs attempt anything else?
    • Attestation: advisory (recommended) vs gate stale/mismatch out?
    • Where does get_online_features live — serving plane (recommended) vs control?
