    Active Versioning — Aliases, Promotion, Shadow, Canary

    Status: DELIVERED (commits 1–3) · Owner: features platform · Created: 2026-06-20

    1. Goal

    Turn the half-built versioning primitives (register_version / get_version, registry.py) into the operational maturity layer a state-of-the-art Feature Platform needs:

    • Aliases — named, indirected pointers (current, previous, stable) so callers reference a role, not a frozen version.
    • Promotion — atomically repoint current to a new version without touching any caller.
    • Rollback — one call back to the prior version (uses previous).
    • Shadow serving — run a candidate alongside current on real traffic, compare outputs, serve only current. (commit 2 — see §6)
    • Canary — route a fraction of traffic to the candidate, with a rollback signal. (commit 3 — see §7)

    This axis consumes the version-drift signal the training generator already emits (TrainingSetReport.stale_plan_hashes): a vector showing drift is a promotion/rollback candidate.

    2. Where versioning is today

    • register_version(name, version, encoder) → compound key name@version.
    • get_version(name, version) / get_all_versions / list_versions.
    • get(name) → the plain-name key = the de-facto "current", but there is no indirection: "current" is simply whatever was last register-ed under the plain name. No alias, no promotion, no rollback, no "previous".

    The semantic_version field (FeatureVectorSettings.semantic_version, surfaced on FeatureCatalogEntry) is the natural version label.

    3. Commit 1 — Aliases + Promotion + Rollback (control plane)

    3.1 Registry: alias indirection

    Extend FeatureVectorRegistry with an alias map { "<vector>:<alias>" → version } plus mutation guarded by the inherited asyncio.Lock.

    def set_alias(self, vector_id: str, alias: str, version: str) -> None
def resolve_alias(self, vector_id: str, alias: str) -> EncoderProtocol
def alias_version(self, vector_id: str, alias: str) -> Optional[str]
async def promote(self, vector_id: str, version: str) -> None   # see 3.2
async def rollback(self, vector_id: str) -> str                 # see 3.3
def current_version(self, vector_id: str) -> Optional[str]      # = alias "current"

    set_alias validates the target name@version is registered (else ValueError) — an alias may never dangle.

    Reserved aliases: current (the serving default) and previous (the prior current, for rollback). Users may add arbitrary aliases (stable, experimental).

    3.2 Promotion (atomic)

    promote(vector_id, version): 1. Validate name@version is registered. 2. Under the lock, atomically: - record the existing current (if any) as previous, - set current → version, - repoint the plain vector_id key to the name@version encoder so the existing serving path (get(vector_id) → encode) reflects the promotion with zero caller changes.

    So promotion is a single control-plane call; the hot path is untouched and needs no awareness of versions.

    3.3 Rollback

    rollback(vector_id): promote(vector_id, alias_version(vector_id, "previous")). Raises if there is no previous. After rollback, previous points to the version that was current (so rollback is itself reversible).

    3.4 Bootstrap wiring

    When a vector bootstraps healthy, register it both ways: register(vector_id, encoder) (current, unchanged) and register_version(vector_id, semantic_version, encoder), then set_alias(vector_id, "current", semantic_version). This makes promotion / rollback meaningful across the semantic versions declared in settings, while keeping get(vector_id) behaviour identical.

    3.5 Facade (control plane)

    On FeaturePlatformService: promote_version(vector_id, version), rollback_version(vector_id), set_version_alias(vector_id, alias, version), list_versions(vector_id) / current_version(vector_id).

    3.6 Tests (commit 1)

    • alias set/resolve; dangling alias rejected.
    • promote repoints current + plain key; get(vector_id) returns promoted encoder; previous records the displaced version.
    • rollback returns to prior; rollback-with-no-previous raises; rollback is reversible.
    • bootstrap registers version + sets current alias.
    • facade delegates.

    4. Failure modes & guards

    Risk Guard
    Alias points at an unregistered version set_alias/promote validate registration → ValueError
    Concurrent promote vs serving read Promotion mutates under the registry asyncio.Lock; get is GIL-atomic dict read
    Rollback with no history rollback raises ValueError when previous is absent
    Promotion to a failed/zombie encoder Only healthy vectors register a version at bootstrap

    5. Original angle

    Promotion/rollback operate on compiled plans with attested plan_hash, so a promotion is a provable, atomic swap of one verified plan for another — and it composes with the training generator: stale_plan_hashes tells you which vectors drifted, promotion/rollback is how you act, and the skew-free generator proves the post-promotion training set matches the newly-served plan.

    6. Commit 2 — Shadow serving (DELIVERED, fully configurable)

    Decisions (all configurable via ShadowServingSettings; recommended = default): - Background + sampled — candidate runs in a fire-and-forget task on a fraction of requests (sample_rate, default 0.05; 1.0 = full coverage). Zero serving-latency impact. The background architecture is fixed (a never-served candidate has no reason to block the hot path); coverage is the knob, not inline-vs-background. - Pluggable metric — DivergenceMetricProtocol + DivergenceMetricRegistry (max_abs default, l2, cosine); add one by registering a class (OCP). Selected by settings.shadow.metric. - Advisory — divergence → feature_shadow_divergence histogram + feature_shadow_comparisons_total{status=ok|flagged|error}; above divergence_threshold → flagged + warning log. Promotion stays a separate manual call. The threshold is recorded so a future promotion gate just reads it — seam ready, behavior not built (no dead config). - Candidate designated by the registry shadow alias (set_version_alias(vector_id, "shadow", version)) — reuses the alias layer.

    Wiring: ShadowServingController built by FeaturePlatformFactory from settings (returns None when disabled → zero hot-path work); injected into FeaturePlatformService, invoked via maybe_observe in encode/encode_batch/ encode_tensor_batch.

    7. Commit 3 — Canary (DELIVERED, fully configurable)

    Decisions (all configurable via CanarySettings; recommended = default): - Sticky-by-entity routing (default) — consistent hash of the routing_key so the same entity always lands on the same version. Pluggable via CanaryRoutingProtocol + CanaryRoutingRegistry (sticky_entity, probabilistic); selected by settings.canary.routing, swap/extend without code change (OCP). Missing routing key → conservatively serves current. - Manual rollback — operators ramp via set_canary(vector, version, fraction) (per-vector fraction, overridable at runtime), graduate via promote_version, abort via clear_canary. feature_canary_requests_total{version,route} is the read-side seam a future automatic gate consumes — behavior not built (no dead config). - Reuses the alias layer — candidate designated by the canary alias; graduation is just promote. Coherent with shadow (shadow alias) + promotion.

    Wiring: CanaryRouter built by FeaturePlatformFactory from settings (returns None when disabled → zero routing overhead); FeaturePlatformService.encode* gained an optional routing_key and routes via _resolve_serving.

    8. Out of scope

    • Multi-pod alias consensus (alias state is per-process today; cross-pod promotion would persist aliases via the catalog repository — follow-up).
