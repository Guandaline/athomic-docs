    SOTA Roadmap — Feature Platform

    Status: LIVING DOC · Owner: features platform · Created: 2026-06-20

    0. Purpose

    The goal is to release this Feature Platform only when it is genuinely state-of-the-art. This document is the map: an honest capability gap analysis versus the reference platforms, the original contributions that already set us apart, a value-ranked sequence of remaining epics, and an objective definition of "SOTA-ready" so the release decision is data-driven, not a feeling. It is a living doc — update statuses as epics land.

    Reference platforms: Uber Michelangelo, Tecton, Feast, Vertex AI Feature Store, Hopsworks, Databricks/Feathr.

    1. Where we already lead (original contributions)

    These are not parity items — they are things most/all reference platforms do not do, enabled by the AOT-compiled single-plan architecture:

    1. Compiled-plan-attested point-in-time correctness (DELIVERED — skew-free-training-generator.md). Training rows are admitted only when their materialization plan_hash equals the live serving plan_hash → training/serving skew eliminated by construction, not by convention. Tecton/Feast run different offline (Spark/SQL) and online (Python) engines and can only mitigate skew.
    2. AOT-compiled feature DAG — CSE, topological tiering, locality optimization, deterministic memory offsets, plan_hash identity. Reference platforms lean on Spark/SQL pipelines; a compiled plan with a cryptographic identity is a distinctive engine design.
    3. plan_hash-gated atomic promotion (DELIVERED — active-versioning.md). Promotion is a provable, atomic swap of one verified compiled plan for another; shadow/canary reuse the same alias machinery.
    4. Scenario-weighted architecture scorecard (PLANNED — architecture-scorecard.md). Objective per-module architecture scoring weighted by module role — not offered by any code-quality tool we surveyed.

    3. Value-ranked epics (the sequence)

    Ranked by SOTA value per effort, consuming seams already built.

    Tier 1 — closes the most-visible SOTA gaps ✅ DELIVERED

    1. Online feature serving (read path) — ✅ get_online_features reads precomputed vectors from the online store (no recompute), with plan_hash attestation (online analog of skew-free training). (online-feature-serving.md)
    2. Freshness SLA monitoring — ✅ per-read age + stale classification + feature_online_age_seconds; delivered with the read path.

    Tier 2 — completes the lifecycle loop

    1. Backfill + recompute-on-mismatch (Model B) — ✅ DELIVERED. training/raw_state.py (raw-state PIT port + in-memory adapter) + training/recompute.py::RowRecomputer (re-runs raw state through the current plan via a narrow encoder port) + TrainingPolicy.RECOMPUTE_ON_MISMATCH: stale/missing rows become recomputed (skew-free, current plan_hash) and enter the matrix; rows with no recoverable raw state stay missing. Backfill of a newly-added feature is the same path (all-missing → recompute). Falls back to attest-only (Model A) when no recomputer is injected. (skew-free-training-generator.md §9.)

    2. Promotion gate + automatic canary rollback — ✅ DELIVERED. In-process VersionHealthTracker (fed by shadow divergence + canary outcomes) gates promote_version and drives automatic canary rollback (background monitor or manual). (version-health-gating.md)

    3. Streaming windowed aggregations — ✅ DELIVERED. Stream Feature View aggregation semantic: ai/features/streaming/aggregation/ — AggregationFunctionProtocol + 8 providers (count/sum/mean/min/max/first/ last/std) + AggregationRegistry (Strategy+Registry), WindowAggregationSpec (config), WindowAggregator (per-entity reduction of a closed window), and WindowAggregationBridge (materializes to any existing sink). All three window types in the shared engine: tumbling ✅; sliding ✅ (SlidingWindow + backward-compatible Window.windows_for() fan-out — an event lands in all ceil(size/slide) overlapping windows); session ✅ (SessionWindow gap-driven, dynamic per-entity boundary; WindowStateManager takes a stateful add path dispatched by a polymorphic is_session, extending the session end on each in-gap event and re-scoring the active index so the trigger closes it after a full gap of silence — reuses the existing storage/lease/trigger primitives, no parallel manager). The aggregator is window-type-agnostic, so session-windowed aggregations flow end-to-end through the same bridge. (The pre-existing WindowStateManager typing debt surfaced on the way was fixed — windowing-state-manager-typing.md.)

    4. Governance: RBAC + approval workflows — access control on read/promote/define, promotion approval gates. Builds on owner/tags.
    5. Feature discovery API — ✅ DELIVERED. control.discovery: search(FeatureQuery) (owner/tag/status/name/dimension/source_key), describe, owners/tags facets, consumers_of(source_key) impact analysis.

    Tier 4 — distribution + objective quality

    1. Multi-pod control-plane consensus — ✅ DELIVERED. Release pointers (alias map current/previous/canary + canary fraction) persist through a ReleaseStateRepository (versioning/release_repository.py: in-memory default
    2. KV-backed RedisReleaseStateRepository). ReleaseStateSynchronizer write- through on every promote/rollback/canary decision; FeatureVectorRegistry.apply_aliases adopts a cluster snapshot (repointing serving, skipping locally-uncompiled versions). A fresh replica reconciles at bootstrap; an opt-in periodic refresh (ReleaseConsensusSettings) converges long-running pods live. Only pointers are shared — compiled encoders stay per-pod.
    3. Architecture scorecard (architecture-scorecard.md) — objective, scenario-weighted module scoring; doubles as the release gate (§4).

    Evaluated and parked

    • CompiledPlanRepository — measured NO-GO for typical fleets (compiled-plan-repository.md §0); revisit only if compile p99 climbs into tens of ms.
    • Control/Serving plane split — organizational refactor; do after Tier 1–2 so each plane's responsibilities are known (training-gen, promotion, canary → control; encode/online-read → serving). The pieces are already pre-isolated.

    4. Definition of "SOTA-ready" (objective release gate)

    Release when all hold (measured, not judged):

    1. Capability parity: every ⚠️/❌ in §2 marked required-for-release is ✅ — minimally Tier 1 (online read + freshness) and Tier 2 (backfill + promotion gate). Streaming/governance (Tier 3) may be explicitly deferred to v1.1 if documented as such.
    2. The differentiators hold under test: the skew-free invariant (byte-identical online vs training) and plan_hash-gated promotion have green end-to-end tests.
    3. Architecture scorecard (§3.9) ≥ target per module profile, with no module regressing below its floor — the objective architecture bar.
    4. Quality gate stays green repo-wide: mypy 0 · pyright 0 · pylint 10/10 · import-linter contracts kept · benchmark p99s within budget.
    5. Operational story documented: materialize → serve-online → monitor freshness/drift → shadow → canary → promote/rollback, end to end.

    5. Non-goals (for the SOTA release)

    • A hosted UI (API/CLI surface is sufficient for v1).
    • Managed cloud control plane / SaaS multi-tenancy beyond per-process + shared catalog/online store.
    • Replacing the compute engine with Spark/SQL — the compiled-plan engine is the differentiator.
