    🧩 Engineering Task: Pluggable AuthorizationProvider — RBAC and ReBAC, switchable by settings

    Status: Done Priority: High Raised by: Nala Ori M4 (Authorization) — consumer-driven gap Extends: rbac-authorization.md (the RBAC engine — Stages 1+2+KV resolver, DONE)

    🎯 Context

    athomic's authorization today is RBAC-only. security/auth/authz ships a solid, pure RBAC engine — AuthorizationService.is_allowed/require, permission_matches (segment-wise wildcards), static/kv permission resolvers, a PermissionResolverRegistry, rollout mode (disabled|dry_run|enforce), authz_decisions_total metrics, and a requires(permission) FastAPI guard. That is the right model for role → permission questions.

    It is not the model the consumer-facing spec calls primary. Nala Ori's Identity & Access §2 (Authorization) specifies Relationship-based access control (ReBAC), default-deny as the primary model:

    Entities: RelationTuple(subject, relation, resource_urn) · built-in RBAC fallback: Role, Permission, Grant · Policy. Ports / adapters: AuthorizationProvider → rebac (OpenFGA, recommended prod), rbac (built-in), abac (Casbin), allow_all (dev no_op). Methods: check, list_objects (list filtering), expand (explainability).

    So there is a two-fold gap:

    1. No ReBAC engine anywhere in athomic — no relation tuples, no check/list_objects/expand over a tuple graph, no OpenFGA (or embedded) adapter. The RBAC engine cannot answer “is user:X an editor of workspace:Y because they’re an admin of its parent org:Z?” — relationship questions that ReBAC/Zanzibar models are built for.
    2. No provider-level abstraction above the engines. RBAC is reachable only as the concrete AuthorizationService; there is no AuthorizationProvider port under which rbac and rebac (and later abac, allow_all) are interchangeable adapters selected by a single setting.

    What the consumer needs (the crux of this task): athomic must provide both authorization models, and a consumer (Nala Ori, and any future solution on the substrate) must be able to switch between them purely via settings — flipping authorization.provider from rbac to rebac with zero changes at the call sites (requires(...) / the provider port). The provider-agnostic seam is the deliverable as much as the ReBAC engine itself.

    This is framework-level platform work and recurs in every web-facing solution on the substrate; it belongs in athomic, not re-implemented per solution.

    ✅ Deliverables

    A. The AuthorizationProvider port (the switchable seam)

    • [ ] An AuthorizationProvider Protocol under security/auth/authz/ exposing the full method set the spec names — a superset of today’s is_allowed/require:

      python class AuthorizationProvider(Protocol): async def check( self, subject: str, relation: str, resource: str, context: Optional[Mapping[str, object]] = None, ) -> Decision: ... # allow/deny (+ reason), default-deny async def list_objects( self, subject: str, relation: str, resource_type: str, ) -> List[str]: ... # list-filtering ("what can X see?") async def expand( self, relation: str, resource: str, ) -> ExpandTree: ... # explainability ("why / who has access?")

      subject/resource are URNs (nala:user:…, nala:workspace:…); relation is the relation/permission token (editor, features:promote). check returns a small Decision value object (allowed: bool, reason/obligations), not a bare bool, so dry_run/expand and audit have a structured result.

    • [ ] An AuthorizationProviderRegistry (Strategy + Registry — mirror the existing PermissionResolverRegistry: BaseRegistry, entry-point plugin group, e.g. nala.auth.authz_providers). register_defaults() registers rbac and rebac; the seam leaves abac and allow_all trivially addable later.

    • [ ] build_authorization_provider(settings, *, kv_store=…, tuple_store=…) factory (sibling to build_authorization_service) that resolves the provider named by settings.provider from the registry, injecting whatever that provider needs, and raising a clear ValueError when a required dependency is absent (same contract as the resolver factory’s kv_store handling).

    B. The rbac provider (adapt the existing engine — no rewrite)

    • [ ] An RbacAuthorizationProvider that wraps the existing AuthorizationService and satisfies the new port:
      • check(subject, relation, resource) → maps to AuthorizationService.is_allowed/require (relation == resource:action permission string; tenant derived from resource/ExecutionContext), honoring mode exactly as today.
      • list_objects / expand → derived from the role→permission mapping where meaningful (enumerate granted permissions / explain via role expansion), or an explicit, documented NotSupported for the dimensions RBAC genuinely cannot answer — never a silent wrong answer.
      • No behavior change for existing RBAC consumers; the concrete AuthorizationService and requires(permission) keep working.

    C. The rebac provider (the new engine)

    • [ ] A relation-tuple model: RelationTuple(subject, relation, resource) (+ optional userset/subject_relation for group-to-group, Zanzibar-style) and a TupleStore port: write(tuples), delete(tuples), read(filter).
    • [ ] A ReBAC engine computing check/list_objects/expand over the tuple graph against an authorization model (object types, relations, and how relations compute — direct, union, tuple-to-userset for hierarchy/inheritance). Default-deny.
    • [ ] Two tuple-store adapters, selectable by sub-config, so the provider works with and without external infra:
      • openfga — recommended for prod; thin client over an OpenFGA server (store-id/model-id/endpoint), delegating check/list_objects/expand and tuple writes to OpenFGA.
      • embedded — a relational/in-memory tuple store + in-process evaluator for dev/CI and single-node deployments, so adopting ReBAC needs no new infra to start. (Add OpenFGA to the dev/CI compose only behind this provider; do not force it on RBAC deployments.)
    • [ ] Tuple write/delete API surfaced through the provider (or a sibling TupleWriter) so a consumer can sync tuples from its domain events (Ori derives tuples/grants from membership.added / team.member_added).

    D. Settings — one switch, provider-agnostic call sites

    • [ ] Extend AuthorizationSettings (security.auth.authorization) with:
      • provider: Literal["rbac", "rebac", "abac", "allow_all"] = "rbac" (default rbac for back-compat; switching is a pure settings change).
      • Orthogonal, shared knobs kept as-is: mode (disabled|dry_run|enforce), require_resource_scope.
      • Provider sub-config (nested, only read by the selected provider): rbac keeps resolver/roles; rebac adds store: "openfga"|"embedded", OpenFGA endpoint/store_id/model_id, and the model/relations source.
    • [ ] The requires(...) dependency and get_authorization_* provider become provider-agnostic: the same route guard and the same composition call resolve to whichever provider settings.provider selects. Flipping rbac → rebac changes no route/call-site code. (The lru-cached provider may become a single composition-root-injected AuthorizationProvider when async lifecycle wiring lands — see rbac-authorization.md §6 note.)

    E. Observability & tests

    • [ ] Extend authz_decisions_total with a provider label ({permission/relation, result, provider}); structured deny logs from both engines (audit parity).
    • [ ] Unit tests for the ReBAC engine (direct, union, tuple-to-userset hierarchy; default-deny; list_objects/expand). Adapter tests for embedded; an OpenFGA adapter test (skippable when no OpenFGA, like other infra-gated tests).
    • [ ] A switch test: the same requires(...)/check call resolves correctly under provider=rbac and under provider=rebac (settings-only change), proving the seam.

    ⚙ Technical Notes

    • One resolution contract, two engines. AuthorizationProvider.check is the single question both engines answer ((subject, relation, resource) → Decision). RBAC answers it by role→permission expansion + wildcard match; ReBAC by walking the tuple graph. Consumers depend only on the port.
    • mode is orthogonal to provider. disabled|dry_run|enforce applies uniformly regardless of which engine is selected — dry_run meters result=would_deny for ReBAC too, so a consumer can roll ReBAC out in dry-run before enforcing, exactly like RBAC today.
    • No new infra forced. The embedded ReBAC store keeps adoption dependency-free; OpenFGA is opt-in for prod. Mirrors the static(no infra) vs kv(redis) resolver split already in place.
    • RBAC stays the fallback. Per the spec, RBAC is the built-in fallback under ReBAC; a consumer may run rebac as primary while RBAC remains available — keep the RBAC engine first-class, do not deprecate it.
    • Tuple derivation is the consumer’s job; the write API is athomic’s. athomic provides write/delete/read; the consumer (Ori) decides which tuples to write from its events. Do not bake domain-event consumption into athomic.
    • Reference consumer to fold in / validate against: nala-ori-api M4 — modules/authz/ (the hexagon), core/authz (the AuthorizationChecker port + requires(...) dep bridging Ori’s cookie Principal → the provider), and the relation_tuples/roles/grants tables (RLS, tenant-scoped). Ori needs the switch to choose rbac for M4.0 and migrate to rebac later without touching its routes.

    🧪 Acceptance Criteria

    • [ ] A consumer selects the authorization model with one setting (authorization.provider) and switching rbac ↔ rebac requires no code changes at any route guard or check call site.
    • [ ] Both providers are real: rbac (wrapping the existing engine, unchanged behavior) and rebac (relation-tuple check/list_objects/expand, default-deny) with an embedded store (no external infra) and an openfga adapter.
    • [ ] mode (disabled|dry_run|enforce) behaves identically under both providers; authz_decisions_total{...,provider} and deny logs are emitted by both.
    • [ ] ReBAC supports hierarchy/inheritance via tuple-to-userset (e.g. org-admin ⇒ workspace-editor) and exposes list_objects for list-filtering and expand for explainability.
    • [ ] athomic exposes a tuple write/delete/read API a consumer can drive from its own events; athomic itself consumes no domain events.
    • [ ] Nala Ori M4 can ship on provider=rbac today and flip to provider=rebac later through settings alone, keeping the same modules.authz routes and the same requires(...) guards.
