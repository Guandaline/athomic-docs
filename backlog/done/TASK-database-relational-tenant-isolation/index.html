    🧩 Engineering Task: Relational tenant-isolation substrate — RLS session-GUC helper for Postgres

    Status: Done Priority: High Raised by: Nala Ori — Organizations & Tenancy (modules/tenancy.md §1) / Security Architecture (tenant isolation) — consumer-driven gap Relates to: multi-tenancy-features.md (feature-data-lake tenancy — the analytical side; this task is the relational side)

    🎯 Context

    athomic already has a complete tenant-context substrate and context-aware isolation for the non-relational stores, but there is a hole exactly where the consumer's system of record lives — the relational database.

    What exists today (verified in src/nala/athomic/):

    • Tenant context propagation — FULL. context/context_vars.py (get_tenant_id() / set_tenant_id()), context/execution.py (ExecutionContext captures tenant_id), and context/propagation.py (capture_context() / restore_context()), the latter already wired into resilience/sagas and workflows so tenant scope survives into background work.
    • Context-aware isolation for Graph / Vector / KV — FULL. Each has a wrapper that transparently injects the tenant from context: database/graph/wrappers/context_aware.py (Cypher param), database/vector/wrappers/context_aware.py (upsert/search/delete filter), database/kvstore/wrappers/key_resolver_kv_client.py (tenant-prefixed keys).

    What is missing:

    • Relational tenant isolation — NONE. database/relational/providers/postgres/client.py sets up the pool and does a SELECT 1 health check, and nothing else touches the tenant. There is:
    • no per-transaction hook that stamps the tenant onto the connection (set_config('nala.org_id', <uuid>, true) / SET LOCAL),
    • no helper to emit the standard Postgres Row-Level Security policy DDL,
    • no ContextAwareRelationalClient sibling to the Graph/Vector wrappers.

    So the one store where a forgotten WHERE organization_id = … is a cross-tenant data breach is the one store with no substrate-level guard. Every consumer on athomic that puts tenant data in Postgres — Nala Ori first, but this recurs for every SaaS solution on the substrate — must currently hand-roll the RLS session-GUC wiring. Ori already has (see ori-rls-guc-nullif-pattern): its RLS policies must NULLIF the nala.org_id GUC because a pooled connection that was never stamped presents '', which explodes on ''::uuid. That footgun is exactly the kind of thing a substrate primitive should own once, correctly, for everyone — not re-discover per consumer.

    Per Nala Ori's own boundary rule (nala-ori-athomic-boundary.md §7, "Gaps go upstream"): a genuinely generic capability the consumer needs but athomic lacks is added to athomic, not hacked into the consumer. Relational RLS wiring is generic platform plumbing — it belongs here.

    Boundary guard (§2, "no SaaS semantics leak down"): this task adds a generic per-connection session-variable / RLS mechanism keyed by the athomic tenant id. It must not learn what an "organization", "plan", or "tenant lifecycle" is. The GUC name and the policy predicate are configuration the consumer supplies; athomic supplies the mechanism.

    ✅ Deliverables

    A. Per-transaction tenant stamping (the core primitive)

    • [ ] A session/connection hook in database/relational/ that, at the start of each transaction (or checkout for a transaction-scoped session), stamps the current tenant onto the connection via a transaction-local GUC:

      sql SELECT set_config(:guc_name, :tenant_id, true); -- true = local to txn

      The tenant is read from context_vars.get_tenant_id() — no argument threading at call sites, mirroring the Graph/Vector wrappers. Implement as a SQLAlchemy begin/checkout event listener (or an async session factory that wraps session.begin()), whichever fits the existing relational provider shape. SET LOCAL/set_config(..., true) so the value dies with the transaction and cannot leak across pooled reuses.

    • [ ] Unstamped-connection safety. When context carries no tenant (system jobs, migrations, pre-auth requests), the hook must leave the GUC in a safe, explicit state — set it to ''/NULL deterministically rather than inheriting a stale value from a previous checkout. Pair this with the policy helper (Deliverable B) so "unstamped" means deny-all, never accidentally-all. Document the NULLIF(current_setting(:guc, true), '') contract so consumers stop re-discovering the pooled-''::uuid footgun.

    • [ ] Configurable GUC name (default e.g. nala.org_id), so athomic owns the mechanism and the consumer owns the name/semantics.

    B. RLS policy helpers (so the DB physically enforces it)

    • [ ] A helper that emits the canonical tenant-isolation policy DDL for a table, for use inside consumer migrations — the mechanism, not the schema:

      python def tenant_rls_policy_sql( table: str, *, tenant_column: str = "organization_id", guc: str = "nala.org_id", policy_name: str = "tenant_isolation", ) -> str: ...

      emitting ENABLE ROW LEVEL SECURITY + a USING (and WITH CHECK) predicate built on NULLIF(current_setting(:guc, true), '')::uuid so an unstamped connection sees zero rows (and cannot write cross-tenant), instead of erroring on ''::uuid. Symmetric WITH CHECK closes the write path too.

    • [ ] Optional FORCE ROW LEVEL SECURITY switch in the helper so table owners are subject to the policy as well (important because app roles are often the table owner).

    • [ ] A documented, auditable bypass path for legitimate system operations (migrations, cross-tenant admin/back-office, maintenance) — e.g. a system_context() / bypass_rls() context manager that runs under a role or GUC state the policy admits, so bypass is explicit and greppable, never the silent default.

    C. Worker / background parity

    • [ ] Guarantee the stamping hook fires for the worker tier too: any relational transaction opened under a restore_context() scope (sagas, workflows, queue consumers) is stamped from the restored tenant, so isolation holds identically for jobs and HTTP requests. Add a test that opens a session inside a restored context and asserts the GUC is set.

    D. Settings & wiring

    • [ ] A multi_tenancy_enabled (or equivalent, aligned with the existing context.multi_tenancy_enabled used by the feature-lake tenancy) switch that gates the stamping hook: on → every transaction is stamped and RLS is in force; off → single-tenant deployments pay nothing. Default must be the safe one for a SaaS substrate.
    • [ ] Wire the hook into the relational provider's session/engine construction so a consumer gets isolation by configuration, not by remembering to install a listener.

    E. Observability & tests

    • [ ] A structured warning/metric when a tenant-scoped transaction runs unstamped while multi-tenancy is enabled (catches "forgot to set context" before it becomes a silent empty-result bug).
    • [ ] Integration tests against real Postgres:
      • stamped connection sees only its tenant's rows; a second tenant's rows are invisible on the same pooled engine;
      • unstamped connection sees zero rows and cannot insert cross-tenant (the NULLIF deny-all contract), and does not raise ''::uuid;
      • SET LOCAL value does not survive into the next checkout of the same physical connection (pool-reuse isolation);
      • worker-tier transaction under restore_context() is stamped correctly;
      • bypass_rls() sees all tenants; leaving it re-isolates.

    ⚙ Technical Notes

    • Mechanism, not policy. athomic supplies: (1) the per-txn GUC stamp from context, (2) the policy-DDL helper, (3) the bypass seam. The consumer supplies: the GUC name, the organization_id column, and which tables are tenant-scoped (in its own migrations). athomic never learns what an org is.
    • Transaction-local is non-negotiable. Connection pooling means a physical connection is reused across tenants; a session-level SET would leak. Use set_config(..., true) / SET LOCAL bound to the transaction.
    • Deny-all on empty, never error, never allow-all. The NULLIF(current_setting( guc, true), '')::uuid predicate is the crux: unstamped ⇒ predicate is NULL ⇒ no rows match ⇒ safe. This is the pattern Ori already had to hand-derive; owning it here means every consumer gets it right by default.
    • Mirror the existing context-aware wrappers. Graph/Vector/KV already do transparent, context-driven tenant injection; the relational hook is the missing fourth member of that family and should feel identical to configure.
    • Reference consumer to validate against: nala-ori-api — the tenancy middleware that resolves TenantContext and sets nala.org_id per request, the worker's per-job equivalent, and its RLS'd tenant-scoped tables. Ori should be able to delete its hand-rolled GUC wiring and RLS-policy boilerplate and adopt the athomic helper with no behavior change.

    🧪 Acceptance Criteria

    • [ ] With multi_tenancy_enabled, every relational transaction (HTTP and worker) is automatically stamped with the context tenant via a transaction-local GUC — no call-site changes.
    • [ ] athomic ships a policy-DDL helper producing the canonical RLS policy with the NULLIF(...)::uuid deny-all-on-empty contract and optional FORCE RLS / WITH CHECK.
    • [ ] An unstamped tenant-scoped transaction returns zero rows, cannot write cross-tenant, and does not raise ''::uuid; a stamped one sees exactly its tenant; pooled reuse never leaks a prior tenant's GUC.
    • [ ] A documented, greppable bypass_rls() / system seam exists for migrations and cross-tenant admin.
    • [ ] A tenant-scoped transaction running unstamped-while-enabled is observable (metric/log), not silent.
    • [ ] Nala Ori can remove its hand-rolled RLS session-GUC wiring in favor of the athomic primitive with no change in isolation behavior.

    🔗 Relationship to existing backlog

    • multi-tenancy-features.md — the analytical / data-lake side of tenancy (parquet partition paths, KV prefixing). This task is the relational / OLTP side (Postgres RLS), the one store that document does not cover. Same context.multi_tenancy_enabled switch should gate both.
    • rbac-authorization.md / TASK-security-authorization-providers.md — authorization (who may act). This task is isolation (which rows physically exist for a connection). Complementary: RLS is defense-in-depth beneath authz — the DB refuses cross-tenant rows even if an authz check is forgotten.
