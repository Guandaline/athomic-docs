    4. Performance: Batch Vector Operations ✅ DONE (2026-06-30)

    Location: nala/athomic/ai/rl/algorithms/contextual_bandit.py Issue: The _dot_product and weight updates use raw Python loops (sum(a * b for a, b in zip...)). Requirement: If state dimensions grow, transition to numpy or vectorized operations to reduce CPU overhead during high-frequency inference, while maintaining the stateless interface.

    Resolution (benchmark-driven, adaptive): A benchmark first disproved the naive fix — swapping the single _dot_product to np.dot is a net loss at every realistic dimension (pure Python beat NumPy through 8192 dims) because the per-call list -> ndarray conversion outweighs the BLAS speedup for list-stored weights. The real lever is batching across actions: computing all per-action Q-values as one action_matrix @ state_vector matmul amortizes the conversion and wins from ~16 actions × 64 dims upward (e.g. 256×256: 2004µs → 1374µs).

    Delivered: - _predict_impl now assembles the per-action weight matrix and computes all Q-values once via _batch_dot (also removing the old double-computation of dot products on the exploit path). _batch_dot is adaptive: a NumPy matmul above numpy_batch_threshold total multiply-adds (num_actions × state_dim), the Python loop below; returns plain floats either way (no NumPy scalar leaks through the port). - _dot_product stays pure Python (documented) — fastest for a single dot, used by _prediction_error (RL-007) and the small-batch fallback. - RLAlgorithmSettings.numpy_batch_threshold (default 256, configurable) governs the switch. - Tests: algorithms/test_unity_contextual_bandit.py::TestContextualBanditBatchedPrediction (NumPy/Python equivalence, correctness, no-mutation on the NumPy path).

    5. Type Safety: Action Schema Validation

    Location: nala/athomic/ai/decision/types.py -> DecisionResult Issue: selected_action is currently typed as Any. Requirement: Introduce a Generic type or a Pydantic validation step to ensure that the action returned by RL algorithms strictly adheres to the schema expected by the ActionProvider.

    [RL-002] Exponential Backoff for Persistence Failures ✅ RESOLVIDO

    Ver nala/athomic/ai/rl/memory/providers/kv_vector.py + nala/athomic/resilience/orchestrator_factory.py.

    Resolução: Toda ida ao backend em KVVectorMemoryProvider — save_weights (KV set), load_weights (KV get), store_experience (Vector upsert) e sample_experiences (Vector search) — passa por self._run(orchestrator, op), que executa a chamada dentro do ResilienceOrchestrator (circuit breaker → retry com backoff exponencial via wait_random_exponential). Retry sozinho contra um backend caído é o anti-pattern de retry storm; o circuit breaker abre e falha rápido, deixando o backend se recuperar. O op embrulha apenas o round-trip de rede; a construção de id/payload (ex.: o record_id do experience) fica fora do retry, de modo que uma tentativa já bem-sucedida server-side não é duplicada.

    Isolamento de falhas: KV (pesos) e Vector (experiências) recebem orchestrators/circuitos distintos (rl_memory_kv / rl_memory_vector), então uma indisponibilidade de um backend abre só o circuito dele e nunca bloqueia o outro.

    Ajuste na resiliência (DRY): o ResilienceOrchestrator não tinha factory — o http/factory montava retry+CB inline na mão. Extraí ResilienceOrchestratorFactory.create(settings=, circuit_name=, ...) (settings injetado, sem get_settings() interno — Factory Rule 1; sem Registry, composição única) para centralizar essa montagem. O módulo backoff/ foi avaliado e descartado: é um sequenciador de delay com estado (wait()/reset()) para loops de polling/consumo, não um wrapper de chamada.

    A cadeia espelha RL-001 mas roteia para a memória: lifecycle._build_memory_resilience (composition root chama a factory por backend) → ReinforcementLearningFactory.create(..., kv_resilience=, vector_resilience=) via **kwargs (diferente de lock_provider, explícito porque vai ao trainer) → RLMemoryFactory.create → KVVectorMemoryProvider.build. Orchestrators None (retry+CB desabilitados/ausentes) → fallback "roda uma vez" com warning explícito. Cobertura: memory/providers/test_unity_kv_vector_resilience.py (roteamento KV vs Vector, fallback, retry real em falha transitória, estabilidade do record_id), resilience/test_unity_orchestrator_factory.py (montagem da factory) + asserts de wiring em test_unity_lifecycle.py.

    [RL-006] Shadow Mode Execution ✅ RESOLVED (2026-06-29)

    The ADE runs a secondary "shadow" policy (typically RL) alongside the primary decision, records its outcome + divergence, but returns only the primary — so a candidate model is validated against live traffic before it controls anything.

    • Config: DecisionSettings.shadow: Optional[ShadowPolicySettings] (enabled, policy_name="rl"), mounted like reward/governance.
    • Wiring: DecisionEngineFactory._build_shadow_policy builds it via DecisionPolicyFactory with the same RL/encoder/normalization deps as the live RL policy but no fallback — a candidate's failures must surface (recorded as a shadow error) rather than silently degrading. Injected into DecisionGraphManager(shadow_policy=…, shadow_policy_name=…).
    • Execution: DecisionGraphManager.execute runs the primary graph and the shadow policy concurrently (asyncio.gather, ~max not sum latency). The shadow branch is fully isolated (_evaluate_shadow never raises — captures the error into the outcome), so primary exceptions propagate unchanged.
    • Result: typed DecisionResult.shadow: Optional[ShadowOutcome] (selected_action, policy_name, confidence, diverged, error) — flows into the existing audit via model_dump(). Observation-only.
    • Metric: ai_decision_shadow_decisions_total{graph_id, outcome} where outcome ∈ agree / diverged / error. Divergence rate gauges promotion readiness.
    • Coverage: tests/unit/.../ai/decision/test_shadow_wiring.py (divergence, agreement, isolated failure, disabled/unconfigured no-op).

    Automated rollback-on-drop trigger (stakeholder 2026-06-30: build BOTH seams). Decomposed into two independent PRs:

    • ✅ Seam A — pre-promote eval gate (deterministic) — DONE (2026-06-30). After an offline batch produces candidate weights, they are scored against the pre-training baseline on a held-out batch and only promoted (saved as active + snapshotted) when they don't regress beyond a tolerance; a regressing candidate is rejected and the baseline keeps serving. No bad weights ever reach traffic.
    • Offline evaluator (algorithm-owned): RLAlgorithmProtocol.evaluate(weights, transitions) -> float (higher=better; negative mean absolute prediction error). BaseRLAlgorithm aggregates; each algorithm implements the read-only _prediction_error (epsilon: Q-lookup; contextual: per-action dot product).
    • Gate: ai/rl/promotion_gate.py — WeightPromotionGate (pure decision object) + PromotionDecision. Promote iff candidate ≥ baseline − tolerance; promotes unjudged when held-out samples < eval_min_samples.
    • Settings: RLPromotionGateSettings (enabled / eval_batch_size / eval_min_samples / max_regression_tolerance) on RLSettings.promotion_gate; built in RLTrainerFactory and injected into the trainer (off by default).
    • Wiring: train_offline captures the baseline, runs the gate before save/snapshot, keeps the baseline + skips snapshot on rejection (priority writes still happen). Metrics ai_rl_promotion_gate_decisions_total{outcome}
      • ai_rl_promotion_gate_score{kind}; also now emits the previously-orphaned ai_rl_weight_snapshots_total on a real snapshot.
    • Refactor: shared Transition moved to neutral ai/rl/types.py so the algorithm (math) layer no longer imports the memory (persistence) layer.
    • Tests: algorithms/test_unity_evaluate.py, test_unity_promotion_gate.py, trainers/test_unity_eval_gate.py.
    • ✅ Seam B — post-deploy reward monitor (reactive) — DONE (2026-06-30). A rolling per-model reward window watches live feedback and auto-rolls-back a model whose post-promotion reward degrades.
    • ai/rl/reward_monitor.py — RLRewardMonitor + a RollbackPort Protocol (DIP; satisfied by RLReleasePlane). Bounded per-model _RewardWindow (LRU). observe(model_id, reward) records + (when armed and enough post-promotion samples) compares the window mean against the baseline and rolls back through the port if it dropped below baseline - max_reward_drop, then disarms (one rollback per promotion). mark_promotion(model_id) captures the pre-promotion mean as the baseline and clears the window.
    • Settings: RLRewardMonitorSettings (enabled / window_size / min_samples / max_reward_drop / max_tracked_models) on RLSettings.reward_monitor, off by default.
    • Wiring: built in ReinforcementLearningService (rollback port = self.releases) and injected into RLTrainingPlane; process_feedback calls observe after learning, trigger_offline_training calls mark_promotion when the cycle promoted (train_offline now returns bool).
    • Metrics: ai_rl_reward_monitor_rollbacks_total{model_id} (auto-only) + ai_rl_reward_monitor_reward_mean{model_id, kind=baseline|current}.
    • State is per-pod (like the replay priority overlay): each pod judges the reward it observes; the rollback writes shared KV so the effect is global.
    • Tests: test_unity_reward_monitor.py, test_unity_training_plane_monitor.py.

    [RL-008] Pydantic Validation for Weights Schema ✅ DONE (2026-06-30)

    Weights flow as opaque Dict[str, Any] JSON blobs through the KV store; nothing guarded their structural integrity across (de)serialization.

    • Task: Use TypeAdapter or structured Pydantic models for algorithm weights to ensure internal structural integrity during serializations.
    • Delivered:
    • algorithms/weights.py — Pydantic schemas EpsilonGreedyWeights (q_table: Dict[str, float], action_counts: Dict[str, int]) and ContextualBanditWeights (bandit_weights: Dict[str, List[float]], with a model_validator rejecting ragged per-action vectors), plus WeightSchemaError(ValueError). extra="allow" preserves the sibling keys the _update_impls carry forward ({**weights, ...}).
