    Athomic Runtime & Platform Architecture Backlog

    Merge note (2026-06-24): This document supersedes and absorbs base_service.md, which covered the same services/base runtime hardening scope as a subset of P0–P1 here. base_service.md was deleted; do not recreate it. The stories below were also corrected against the current code state — several "create X" stories described work that partially already exists; those are now phrased as "harden/complete X" with a pointer to what's already implemented.

    Vision

    Transform Athomic from an infrastructure framework with isolated lifecycle and plugin capabilities into a fully observable, dependency-aware, runtime-managed platform while preserving its core principles:

    • Explicit over Implicit
    • Constructor Injection over Service Locator
    • Composition Root over Runtime Magic
    • Predictable Lifecycle Management
    • Operational Transparency
    • Testability
    • Framework Extensibility

    P0 — Foundation Hardening

    These initiatives address architectural debt and runtime correctness. They should be completed before introducing major platform capabilities.

    EPIC-01: Eliminate Global Service State

    Objective

    Remove all framework-level mutable global state and centralize runtime ownership.

    Goals

    • Remove global singleton instances.
    • Improve test isolation.
    • Enable multiple Athomic runtimes per process.
    • Make ownership explicit.

    EPIC-02: Service Lifecycle State Machine

    Objective

    Formalize service lifecycle behavior.

    Current state (corrected)

    ServiceState already exists at src/nala/athomic/services/enums.py (PENDING, CONNECTING, READY, FAILED, CLOSED, RUNNING) and is used throughout BaseService (src/nala/athomic/services/base.py). What is missing is not the enum — it's transition validation: self._state = ServiceState.X is assigned freely from multiple methods (connect(), close(), set_running(), _handle_connection_failure()) with no guard preventing an invalid jump (e.g. CLOSED → READY). Stories below target that gap, not the enum itself.

    Stories

    Story 2.1

    Centralize all _state writes behind a single transition method (e.g. _transition_to(new_state)) instead of direct assignment scattered across connect()/close()/set_running()/_handle_connection_failure().

    Story 2.2

    Define the allowed-transition table and enforce it in _transition_to.

    PENDING    → CONNECTING
CONNECTING → READY | FAILED
READY      → RUNNING | CLOSED
RUNNING    → READY | CLOSED
FAILED     → CONNECTING (retry) | CLOSED
CLOSED     → (terminal)

    Story 2.3

    Raise on invalid transitions instead of silently overwriting _state.

    Story 2.4

    Track lifecycle metadata alongside the existing _last_start_error.

    started_at
failed_at

    Story 2.5

    Expose service diagnostics (transition history / last N transitions) for debugging — health() today only exposes the current snapshot.

    P4 — Platform Consolidation

    EPIC-18: Registry Framework Consolidation

    Goals

    • Standardize all registries.
    • Consolidate APIs.
    • Add type safety.

    Known concrete issues to resolve as part of this epic

    Story 18.1 — Dead/duplicate plugin-discovery channel

    BaseRegistry (src/nala/athomic/base/registry.py) implements two separate plugin mechanisms that overlap with PluginManager (src/nala/athomic/plugins/manager.py):

    1. _discover_plugins() — per-registry entry-point scanning via its own plugin_group, independent of PluginManager.
    2. load_external_plugins(plugin_manager, hook_name) — designed to broadcast a registry to the PluginManager hook ecosystem for IoC injection, but it is never called anywhere in the codebase (verified via repo-wide grep). It is dead code today.

    Decide and implement one canonical path: either wire load_external_plugins into the actual registries that need plugin-injected providers (and document which one), or remove it and rely solely on _discover_plugins() / PluginManager.discover_and_load(). Don't leave both as a maybe-works path.

    Story 18.2 — FactoryProtocol / CreatorProtocol need an explicit, documented distinction

    src/nala/athomic/base/factory.py::FactoryProtocol (classmethod, create(settings, **kwargs) -> T) and src/nala/athomic/base/creator.py:: CreatorProtocol (instance method, same signature) look identical in shape. Usage audit (2026-06-24) shows they are not accidental duplication — both are real, widely used, and serve different roles:

    • FactoryProtocol (~26 call sites, e.g. KVStoreFactory, LLMFactory) — stateless composition-root factory, called once to build a component.
    • CreatorProtocol (~45 call sites, e.g. creators/router_creator.py, creators/static_creator.py) — a polymorphic instance held inside a BaseRegistry, used as the Strategy half of the Strategy+Registry pattern (the registry holds creator instances, not classes).

    Given the size of both call-site sets, collapsing them is not worth the ~70-file refactor. Instead: add a one-paragraph doc-comment on each protocol cross-referencing the other and stating which one to reach for (composition root vs. registry-housed strategy), so the next person doesn't reintroduce a third near-identical protocol out of confusion.

    Success Criteria

    Upon completion:

    • No global mutable framework state exists.
    • Service lifecycle is formally defined.
    • Background tasks are centrally managed.
    • Startup and shutdown are dependency-aware.
    • Health and readiness are independent concepts.
    • Runtime diagnostics are first-class features.
    • Plugins are versioned, validated, and dependency-aware.
    • Athomic supports multiple runtimes per process.
    • Framework observability is production-grade.
    • The architecture remains explicit and free from dependency-injection magic.
