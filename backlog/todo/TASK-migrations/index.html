    Benchmark e Backlog: Sistema de Migrações Athomic

    Data: 12 de Setembro de 2025 Autor: Nala Assistance Status: Análise Concluída, Backlog Definido

    1. Introdução

    Este documento apresenta uma análise comparativa (benchmark) entre a biblioteca yoyo-migrations e a solução customizada de migrações que projetamos para o framework Athomic. O objetivo é validar nossa abordagem e identificar os pontos-chave a serem implementados para garantir uma ferramenta de alta qualidade, alinhada com os princípios do Athomic.

    2. Benchmark de Features: yoyo-migrations vs. Athomic Migration Runner

    A biblioteca yoyo-migrations foi usada como referência de mercado para um orquestrador de migrações agnóstico e baseado em scripts Python.

    Feature yoyo-migrations Athomic Migration Runner (Nosso Design) Vantagem Athomic
    Rastreamento de Migração Usa uma tabela dedicada (_yoyo_migrations) para rastrear scripts executados. Usa uma coleção Beanie (MigrationStatus) com suporte nativo a source para múltiplos conjuntos de migrações. ✅ Superior
    Suporte a Bancos Agnóstico. Requer backends específicos para cada tipo de banco (SQL, SQLite). Não possui backend nativo para NoSQL. Agnóstico através do MigrationBackendProtocol. O MongoBackend é a primeira implementação. O SqlYoyoBackend (adaptador) permite usar o motor do yoyo para SQL. ✅ Superior
    Tipo de Script Suporta arquivos .sql e .py (com funções step ou upgrade/downgrade). Foco em .py com upgrade/downgrade. Isso permite que as migrações usem outros componentes do Athomic (config, logger, etc.) de forma nativa. ✅ Mais Integrado
    Suporte asyncio Sim, possui suporte nativo para operações assíncronas. Nativo e Async-first. Toda a arquitetura foi projetada do zero para ser assíncrona. ✅ Igual/Superior
    Interface de Comando (CLI) Fornece uma CLI completa e robusta (apply, rollback, show, etc.). Projetada com Typer para ser uma CLI unificada (athomic db ...), oferecendo uma experiência de usuário mais coesa e integrada. ✅ Superior
    Gestão de Transações Excelente para SQL. Executa cada migração em uma transação, garantindo atomicidade. Ponto de Melhoria para MongoDB. A atomicidade precisa ser gerenciada manualmente nos scripts de migração que realizam múltiplas operações. Para SQL, o adaptador para yoyo herda essa funcionalidade. 🟡 Inferior (Mongo)
    Hooks Pós-Execução Permite configurar "post-apply hooks" para executar código Python após as migrações. Solução mais robusta via Event Bus. O runner publicará um evento (athomic.migrations.applied), permitindo que qualquer serviço reaja de forma desacoplada. ✅ Superior
    Manutenção Gerenciada pela comunidade de código aberto. Gerenciada pela equipe do Athomic, garantindo total alinhamento com o framework. ⚪️ Trade-off

    Conclusão do Benchmark

    A solução customizada para o Athomic, embora demande um esforço inicial de implementação, se mostra superior em termos de integração, consistência e flexibilidade para o ecossistema do framework. Ela adota os melhores conceitos do yoyo-migrations e os eleva ao integrá-los nativamente com os outros primitivos do Athomic (configuração, eventos, CLI unificada).

    3. Backlog de Implementação

    Com base na análise e no design que finalizamos, aqui está um backlog simples com as tarefas necessárias para construir o módulo de migração.

    Épico: EPIC-007: Implementar Módulo de Migração de Banco de Dados Agnóstico

    Tarefas de Implementação (Core)

    • [ ] TASK-MIG-001: Criar o MigrationBackendProtocol para definir a interface dos backends.
    • [ ] TASK-MIG-002: Implementar o schema Beanie MigrationStatus para rastreamento no MongoDB.
    • [ ] TASK-MIG-003: Implementar a classe MongoBackend que satisfaz o protocolo.
    • [ ] TASK-MIG-004: Criar o migration_backend_registry e registrar o MongoBackend.
    • [ ] TASK-MIG-005: Implementar a MigrationBackendFactory com a lógica de resolução de configuração (inline vs. config_ref).
    • [ ] TASK-MIG-006: Implementar o MigrationRunner que orquestra o processo lendo as sources e usando a factory.
    • [ ] TASK-MIG-007: Adicionar a publicação do evento athomic.migrations.applied no Event Bus ao final do processo de upgrade.

    Tarefas da CLI

    • [ ] TASK-MIG-008: Estruturar a CLI athomic db com Typer no diretório helpers/athomic_cli/commands/.
    • [ ] TASK-MIG-009: Implementar o comando athomic db upgrade.
    • [ ] TASK-MIG-010: Implementar o comando athomic db status com a tabela formatada (rich).
    • [ ] TASK-MIG-011: Implementar o comando athomic db create para gerar templates de migração.

    Melhorias e Próximos Passos (Backlog Futuro)

    • [ ] TASK-MIG-012: Implementar a lógica de downgrade no runner e na CLI.
    • [ ] TASK-MIG-013: Documentar o padrão de uso de transações do Motor dentro dos scripts de migração para MongoDB.
    • [ ] TASK-MIG-014: (Spike) Implementar o SqlYoyoBackend como prova de conceito para o suporte a bancos relacionais.
