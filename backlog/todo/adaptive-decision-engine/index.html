    Engineering Plan: Adaptive Decision Engine & RL Core

    1. Objective

    Implement a highly cohesive, loosely coupled decision-making infrastructure for the athomic.ai framework, explicitly separating the Reinforcement Learning (RL) mathematical core from the Adaptive Decision Engine (ADE) orchestration layer, as defined in ADR-135.

    2. Architecture Overview

    src/nala/athomic/ai/
├── rl/                     # Pure Learning Engine
│   ├── algorithms/         # Epsilon-Greedy, Bandits
│   ├── trainers/           # Online/Offline updaters
│   └── memory/             # State-Action-Reward storage
└── decision/               # Adaptive Decision Engine (ADE)
    ├── policies/           # Core Abstraction (DecisionPolicyProtocol)
    │   └── providers/      # Heuristic, RuleBased, RLPolicy (Adapter)
    ├── features/           # State Encoding
    ├── actions/            # Dynamic Action Space Resolution
    └── governance/         # Safety and Business Constraints

    3. Implementation Phases

    Phase 1: Core Abstractions & Infrastructure (ADE Foundation)

    Establish the central contracts that will dictate how components interact without implementing the underlying logic.

    • 1.1. Define DecisionPolicyProtocol: Create the central interface in decision/policies/protocol.py.
    • 1.2. Setup Registry & Factory: Implement policy_registry and PolicyFactory adhering to the athomic.ai lifecycle.
    • 1.3. Define Auxiliary Protocols: * FeatureEncoderProtocol (decision/features/protocol.py)
      • ActionProviderProtocol (decision/actions/protocol.py)

    Phase 2: Reinforcement Learning Core (Pure Learning)

    Build the isolated mathematical engine. It must not import anything from decision/, router/, or llm/.

    • 2.1. Define RLAlgorithmProtocol: The contract for updating weights and selecting actions based on probabilities.
    • 2.2. Implement Basic Algorithms: Develop EpsilonGreedyAlgorithm and ContextualBanditAlgorithm inside rl/algorithms/.
    • 2.3. Implement Memory Interfaces: Define KV and Vector storage protocols inside rl/memory/ to track (State, Action, Reward) tuples.

    Phase 3: Policy Providers & The RL Adapter

    Connect the learning engine to the decision orchestration layer using the Adapter pattern.

    • 3.1. Implement HeuristicPolicyProvider: A baseline static policy for immediate use.
    • 3.2. Implement RLPolicyProvider: The adapter that implements DecisionPolicyProtocol but delegates the mathematical selection to an injected RLAlgorithmProtocol.
    from typing import Any, List, Protocol

class DecisionPolicyProtocol(Protocol):
    """
    Defines the standard contract for decision-making policies within the ADE.
    """
    async def select_action(self, state: Any, actions: List[Any]) -> Any:
        ...

class RLPolicyProvider(DecisionPolicyProtocol):
    """
    Adapter that integrates the Reinforcement Learning core into the Adaptive Decision Engine.
    """
    def __init__(self, rl_algorithm: Any) -> None:
        self._rl = rl_algorithm

    async def select_action(self, state: Any, actions: List[Any]) -> Any:
        return await self._rl.select_action(state, actions)

    Phase 4: Governance & Safety Pipeline

    Implement the layers that filter actions before they reach the policy and validate the output before execution.

    • 4.1. Action Filtering: Implement GovernanceActionFilter to dynamically reduce the actions list based on context (e.g., removing a premium LLM for a free-tier user).
    • 4.2. Output Validation: Implement SafetyValidator to ensure the action selected by the policy is safe to execute.

    Phase 5: Domain Integration (Router & Guardrails)

    Inject the ADE into the existing athomic.ai modules to replace static heuristics.

    • 5.1. Router Integration: * Create RLRoutingStrategy implementing RoutingStrategyProtocol.
      • Inject PolicyFactory.create("rl_policy").
    • 5.2. Guardrail Integration: * Use a policy provider to dynamically decide the enforcement action (allow, sanitize, block) based on state embeddings.

    Phase 6: Delayed Rewards & Credit Assignment

    Implement the asynchronous loop required for production RL, where actions and rewards occur at different times.

    • 6.1. Event Correlation: Implement a TraceCorrelator in decision/reward/ to link a request_id from the action phase to a metric/feedback received later.
    • 6.2. Background Updater: Implement a RunnableServiceProtocol worker to process logs and update the RL core offline.

    4. Success Criteria

    1. Zero Domain Leakage: The rl/ module imports nothing from router/ or decision/.
    2. Swappable Policies: The Router functions identically whether injected with HeuristicPolicyProvider or RLPolicyProvider.
    3. Strict Typing: mypy passes strictly across all new protocols and providers.
    4. Resilience: If the RLPolicyProvider fails, the system safely falls back to a default deterministic provider without crashing the request chain.
