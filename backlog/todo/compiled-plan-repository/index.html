    CompiledPlanRepository — Cross-Pod / Cross-Restart Plan Reuse

    Status: PLANNED · Owner: features platform · Created: 2026-06-19

    0. Gate-0: measure before building (do this first)

    This subsystem reuses only the graph-analysis passes — milliseconds for small vectors — while the dominant cold-start costs (transformer wiring, model loading) are out of scope and unavoidable on a cold pod. Bootstrap already runs vectors concurrently. So measure the actual compile budget before committing ~5 commits + a Redis subsystem to maintain:

    1. Add a feature_graph_compile_seconds histogram (labels: vector_id, n_features, n_tiers) around compile_plan().
    2. Micro-benchmark compile across representative vectors (1 / 10 / 100 features, shallow vs deep DAGs).

    Go/no-go: if p99 compile is sub-millisecond, the Redis RTT (~1ms) + codec + maintenance likely exceed the saving — stop after the metric and revisit. If deep-DAG vectors compile in tens of ms, or the fleet is large enough that cold pods recompile thousands of vectors serially-enough to matter, proceed. The metric is independently useful regardless of the go/no-go.

    Gate-0 RESULT (measured 2026-06-20)

    feature_graph_compile_seconds shipped + benchmark tests/benchmark/nala/athomic/ai/features/test_benchmark_feature_compile.py. Median compile_plan() on this machine:

    Shape Median
    1 feature 30 µs
    10 features (wide, 1 tier) 138 µs
    10 features (deep, 10 tiers) 108 µs
    50 features (deep, 50 tiers) 430 µs
    100 features (wide) 1.18 ms

    Verdict: NO-GO for typical fleets. A cold pod recompiling a ≤10-feature vector costs ~30–140 µs — cheaper than a single Redis round-trip (~0.5–1 ms) to fetch the persisted plan. The repository would make cold start slower, not faster, for normal vectors. Compile only reaches ~1.2 ms at 100 features.

    Proceed to commits 1–5 only if the production fleet has many large (100+ feature) vectors and cold-pod startup recompiling the whole fleet is a measured bottleneck (bootstrap already runs vectors concurrently, which further shrinks the win). Until then, the histogram stays as a standing monitor; revisit if feature_graph_compile_seconds p99 climbs into the tens-of-ms range in prod.

    1. Goal

    Let a cold pod (empty FeatureVectorRegistry) skip the AOT compiler's graph-analysis passes by reconstructing a previously-compiled FeatureExecutionPlan from a shared store, instead of recompiling from scratch.

    This closes the gap the second architect review flagged: today's incremental bootstrap keys on registry.has(vector_id), so a fresh pod / rolling deploy always recompiles. With this repository, the graph passes are reused across pods and restarts.

    2. Honest scope — what this DOES and does NOT save

    Grounded in the actual code, not the review's assumptions:

    Cost Where it lives Reused by this repo?
    Graph analysis: prune, validate-deps, CSE, topo-sort, locality, offset map, dimension asserts, hash compiler/core.py:78-137 ✅ Yes — this is the win
    Instruction synthesis (_compile_tiers) compiler/core.py:411 ⚠️ Re-run, but cheap — needs freshly-wired imputers
    Transformer/imputer wiring (TransformerWiring.build) encoders/wiring.py:70 ❌ No — transformers are live objects, must re-wire
    Model weight loading (e.g. embedding_manager.get_client) transformations/providers/text.py:47 ❌ No — separate layer, name-keyed, already shared

    Why instructions can't simply be persisted whole: BaseInstruction is a frozen dataclass holding a live imputer object (engines/instruction/base.py:51). We persist its metadata and re-bind freshly-wired imputers on reconstruct.

    Net benefit: the graph passes (milliseconds for small vectors; meaningful for deep DAGs / large fleets). Transformer wiring and model loading are unavoidable on a cold pod regardless — they are out of scope here and already optimized elsewhere (model clients are name-keyed and shared across vectors).

    3. The keying problem (and its solution)

    plan_hash is generated from vector_settings + dimension + offsets (_generate_plan_hash), and dimension/offsets only exist after compilation. So we cannot look up a plan by plan_hash without first compiling — chicken-and-egg.

    Solution: key the repository by settings_fingerprint (already computed cheaply from settings alone in bootstrap.py), store the full plan, and use plan_hash only as an integrity check after reconstruction.

    settings_fingerprint  ──lookup──▶  persisted { envelope + tier_manifest }
                                          │
                          reconstruct (wire imputers, rebuild tiers)
                                          │
                          verify reconstructed.plan_hash == persisted.plan_hash

    4. Persisted artifact (revised — manifest carries only graph-pass output)

    Design principle (from review): the reconstructor already holds the current vector_settings (it is wiring transformers from them). So re-derive everything that lives in settings — original source_keys order, transformation / feature_type, per-transformer dimension — and persist only what the graph passes produced and settings does not contain:

    1. Tier partition — the topo-sort + locality result: which survivor output_names execute in which tier.
    2. CSE alias map — alias_name → survivor_output_name (from optimizer.optimize, applied by _map_cse_aliases).
    {
  "schema_version": "1.0",
  "settings_fingerprint": "ab12cd34ef56...",   // lookup key
  "envelope": { /* existing to_json(): plan_hash, vector_id, dimension,
                   bytes_per_row, offsets, compilation_timestamp */ },
  "tier_partition": [                           // survivor output_names per tier
    ["amount_scaled", "limit_scaled"],
    ["risk_ratio"]
  ],
  "cse_aliases": {                              // alias -> survivor
    "amount_normalized": "amount_scaled"
  }
}

    Why the original manifest was wrong (review §2/§3): - It stored routing_key, which for composites is tuple(sorted(source_keys)) (get_routing_key). But CompositeInstruction._transform_primary does column_stack([views[k] for k in self.source_keys]) over the original unsorted order set by _build_instruction. Deriving source_keys from the sorted routing key would reorder columns → not byte-identical. Re-deriving the FeatureDefinitionSettings per output_name from current settings yields the correct original order. - feature_type (f_def.transformation or transformer.__class__.__name__, core.py:448) was also absent and is not covered by plan_hash — re-deriving from settings fixes that gap too. - The CSE alias map was entirely missing, so reconstruct would lose the alias→survivor offset sharing.

    The manifest is produced at compile time directly from the compiler's own locality_tiers + aliases (the literal graph-pass output), not reverse-engineered from the finished plan — the plan object discards the alias map. This needs compile_plan to return the manifest alongside the plan (a small CompiledArtifact(plan, tier_partition, cse_aliases) return, or a sibling accessor). See §5.2.

    5. Components (SOLID / Hexagonal)

    5.1 Port — CompiledPlanRepositoryProtocol

    New file engines/plan_repository.py. Mirrors CatalogRepositoryProtocol.

    @runtime_checkable
class CompiledPlanRepositoryProtocol(Protocol):
    async def save(self, persisted: PersistedPlan) -> None: ...
    async def get(self, fingerprint: str) -> Optional[PersistedPlan]: ...
    async def remove(self, fingerprint: str) -> None: ...
    async def all_fingerprints(self) -> set[str]: ...

    PersistedPlan is a frozen dataclass: fingerprint, envelope_json (the existing to_json() string), tier_partition: list[list[str]], cse_aliases: dict[str, str], schema_version.

    • MemoryCompiledPlanRepository — default, in-process (no cross-pod benefit; parity + tests).
    • RedisCompiledPlanRepository — key athomic:compiled_plan:<fingerprint>, index set athomic:compiled_plan:__index__ (same pattern as RedisCatalogRepository). Last-writer-wins. KV store typed via a KVStoreProtocol, not Any — and the pre-existing RedisCatalogRepository(kv_store: Any) (catalog_repository.py:207) is a CLAUDE.md Any-ban violation to fix in the same pass (commit 5).

    5.2 Serializer — TierManifestCodec

    Pure, no I/O. Builds PersistedPlan from the compiler's graph-pass output (not from the finished plan, which discards the alias map):

    • compile_plan returns a CompiledArtifact(plan, tier_partition, cse_aliases) — tier_partition from locality_tiers (survivor output_names per tier), cse_aliases from the optimizer's aliases. Existing callers that want only the plan read .plan.
    • codec.to_persisted(fingerprint, artifact) -> PersistedPlan
    • codec.parse(raw) -> PersistedPlan (Redis JSON round-trip).

    No routing keys are stored — they are re-derived from settings on reconstruct using the same get_routing_key rule.

    5.3 Reconstructor — PlanReconstructor

    The core. Given PersistedPlan + current vector_settings + freshly-wired transformers + imputers:

    1. Index settings: {f.output_name: f for f in settings.features} (gives original source_keys order, transformation, etc.).
    2. For each tier in tier_partition, for each survivor output_name: re-derive f_def from settings → routing key → look up live imputer + transformer → output_slice = envelope.offsets[output_name] → build instruction via the shared InstructionFactory (extracted from _build_instruction so compiler and reconstructor share one synthesis path — DRY + single LSP contract). Original source_keys order is preserved → byte-identical.
    3. Apply cse_aliases: for each alias → survivor, the plan's offsets already carries the alias slice (persisted in the envelope); no instruction is created for aliases (matches compiler behavior).
    4. FeatureExecutionPlan.from_json(envelope_json, execution_tiers=rebuilt).
    5. Integrity gate (revised, see §6): recompute _generate_plan_hash(current_settings, persisted.dimension, persisted.offsets) and compare to envelope.plan_hash. On mismatch → log + fall back to full compile. (Comparing from_json's copied plan_hash to itself is vacuous — the only real gate is recomputing from current settings + persisted structure, which catches corruption / schema drift.)

    5.4 Compiler integration — async I/O stays in the coordinator

    FeatureEncoderFactory.create remains synchronous. The repository I/O lives in the async VectorBootstrapCoordinator, exactly mirroring how catalog_repository.load_fingerprints() is already called there (bootstrap.py) and never in the sync factory.

    # VectorBootstrapCoordinator._bootstrap_one (async):
fp = compute_settings_fingerprint(vector_settings)
persisted = await self._plan_repo.get(fp)         # I/O here, async, coordinator
artifact = FeatureEncoderFactory.create(          # SYNC — no await inside
    vector_settings=...,
    persisted_plan=persisted,                     # pre-fetched, may be None
    ...
)
if persisted is None:                             # freshly compiled → persist
    await self._plan_repo.save(
        codec.to_persisted(fp, artifact)          # I/O here, async, coordinator
    )

    Inside sync create: if persisted_plan is not None: PlanReconstructor.rebuild(...) (pure CPU, no I/O) else FeatureExecutionCompiler.compile_plan(...). Transformer wiring happens before this branch (unavoidable), so reconstruct only skips graph passes.

    5.5 Bootstrap / factory wiring

    • VectorBootstrapCoordinator and FeaturePlatformFactory.create() accept compiled_plan_repository: Optional[...] = None and thread it through (same pattern just used for catalog_repository).
    • Default None → MemoryCompiledPlanRepository (zero behavioral change for existing single-pod deployments).

    6. Failure modes & guards

    Risk Guard
    Persisted artifact corrupt / structurally drifted Recompute _generate_plan_hash(current_settings, persisted.dimension, persisted.offsets) ≠ envelope.plan_hash → full compile. Not comparing from_json's copied hash to itself (vacuous).
    schema_version mismatch Treat as miss → recompile + overwrite
    output_name in tier_partition absent from current settings (fingerprint collision / drift) KeyError on settings re-derive → caught → recompile (Error-Kernel isolation, per vector)
    Redis down get() raises → caught → full compile (repo is an optimization, never a hard dependency)
    Reconstruct cheaper than Redis RTT for tiny vectors Optional min_features_for_reuse threshold in settings (skip repo for trivial vectors)

    Scope note on the gate: because settings_fingerprint and the features component of plan_hash both derive from the same model_dump, a fingerprint hit already implies settings identity. The integrity gate's real job is therefore corruption / schema-drift defense, not "settings changed" detection — §6's earlier framing oversold it.

    7. Tests

    • MemoryCompiledPlanRepository: save/get/remove/index round-trip.
    • TierManifestCodec: univariate + composite + fallback records round-trip; routing-key derivation matches get_routing_key.
    • PlanReconstructor:
    • reconstructed plan_hash == original.plan_hash (the round-trip invariant).
    • reconstructed plan executes to byte-identical output vs freshly-compiled plan on a fixture batch (the real correctness proof).
    • integrity-gate: tampered envelope offsets → reconstruct rejected → compile.
    • Factory integration: cold registry + repo hit → compiler graph passes not invoked (assert via patch on FeatureExecutionCompiler.compile_plan), tiers still executable.
    • RedisCompiledPlanRepository: integration-marked, fake KV store.

    8. Estimated shape

    • Commit 0 (gate): feature_graph_compile_seconds histogram + benchmark. Decide go/no-go from real numbers (§0). Independently useful.
    • Commit 1: extract shared InstructionFactory from _build_instruction (compiler refactor, no behavior change); compile_plan returns CompiledArtifact.
    • Commit 2: CompiledPlanRepositoryProtocol + MemoryCompiledPlanRepository
    • TierManifestCodec + PersistedPlan.
    • Commit 3: PlanReconstructor (settings re-derive + integrity gate) + byte-identity tests.
    • Commit 4: sync create accepts persisted_plan; async coordinator pre-fetch/save wiring.
    • Commit 5: RedisCompiledPlanRepository with a typed KVStoreProtocol (and fix the pre-existing RedisCatalogRepository(kv_store: Any) violation in the same pass); factory wiring + docs.

    Each behind the quality gate (mypy 0 · pyright 0 · pylint 10 · tests green).

    9. Explicitly out of scope (separate efforts)

    • Model-weight warm cache — the genuinely expensive cold-start. Belongs to the embedding/connection layer, not the plan. Already name-keyed & shared.
    • Control-plane / serving-plane split of FeaturePlatformService — the review's other headline; a larger facade redesign tracked separately.
