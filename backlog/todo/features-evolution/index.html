    [FEAT-012] Extended Fallback Chains & Matrix Imputation

    Blocked by: Core ML Module (required for KNN state persistence and vectorized distance calculations).

    Tasks: - Implement ForwardFillTransformation (requires temporal matrix context). - Implement KNNImputationTransformation (requires ML distance metrics and fitted state). - Create FallbackChainTransformation orchestrator ([primary_transform -> fallback_transform -> final_scalar_impute]).

    Acceptance Criteria: Graceful degradation under data quality issues without breaking the columnar float32 memory bounds or the strict execution engine contracts.

    🟡 Strategic (pursue only when concrete trigger is identified)

    Auto-Tuning A — Adaptive Batch Splitting

    Trigger: Only pursue if a consumer is observed sending batches that cause p99 encoding latency to exceed SLA, and the caller has no control over batch sizing.

    Approach: Intercept in BaseFeatureEncoder.encode_batch() / FeatureEncoderService.encode_tensor_batch(). If batch_size > adaptive_split_threshold (configurable), split the batch into chunks, execute each through the engine independently, and concatenate results before returning. The threshold should be derived from the feature_encoding_duration_seconds p99 histogram, not hardcoded.

    What NOT to do: Do not implement a dynamic threshold that adjusts itself at runtime — that creates oscillation. Use a static threshold set via config, informed by offline analysis of the latency histogram.

    Acceptance Criteria: - [ ] A specific consumer with demonstrable SLA breach is identified first. - [ ] Split is transparent to callers — same response shape regardless of whether splitting occurred. - [ ] A config flag adaptive_batch_split_threshold in FeatureVectorSettings enables/disables it per vector. - [ ] Splitting adds at most one extra allocation per chunk (no copies beyond what the buffer pool already handles).

    Auto-Tuning B — Engine Migration Based on Observed Throughput

    Trigger: Only pursue if a deployment runs both AsyncCPUEngine and TorchEngine and there is evidence that the wrong engine is selected at the feature-vector level due to static config.

    Approach: After N encoding calls, compare feature_encoding_duration_seconds p50 for the configured engine against a shadow execution on the alternative engine. If the alternative consistently outperforms by more than a configurable margin, emit a warning log recommending a config change. Do NOT auto-migrate at runtime — engine selection affects memory layout and compilation graphs; silent migration would be a correctness risk.

    Acceptance Criteria: - [ ] A concrete deployment with mixed engine types and measurable latency difference is identified. - [ ] Comparison is based on observed feature_encoding_duration_seconds, not synthetic benchmarks. - [ ] Output is a structured log warning with vector_id, current engine, recommended engine, and observed delta — never a silent config change.

    [ARCH-04] RayEngine — True Distributed Task Dispatch

    Trigger: Only pursue when a workload requiring cross-node feature computation is identified. Single-node GPU + NumpyBufferPool covers most real workloads.

    Status: RayEngine (engines/providers/ray_engine.py) exists but does not use Ray for distribution. Its _execute_tensor_batch_impl uses asyncio.gather identically to AsyncNumpyEngine — Ray is only initialized for the connection, not for dispatching work.

    What true distributed execution requires: - ExecutionNodeProtocol implementations must be serializable (pickle-compatible) for Ray's object store - ColumnarFeatureContext columns must be transferable via ray.put() / shared memory - Tier execution must use ray.remote + ray.get() instead of asyncio.gather - Cross-node result aggregation and memory offset merging

    Blocking conditions: 1. A use case requiring cross-node feature computation must be identified 2. Ray cluster infrastructure must be provisioned and tested 3. Instruction serialization strategy must be defined — frozen dataclasses are pickle-friendly, but transformer objects they hold may not be

    Acceptance Criteria (when unblocked): - [ ] A workload is identified that genuinely requires cross-node execution. - [ ] ColumnarFeatureContext verified pickle-compatible via ray.put(). - [ ] RayEngine._execute_tensor_batch_impl uses ray.remote for at least one tier. - [ ] Fallback to asyncio.gather when Ray cluster is unavailable.

    Feature Importance Scoring

    Blocked by: A downstream ML model that consumes the feature vector (feature importance is always relative to a specific prediction target — without knowing what model uses the feature vector, there is no importance to compute).

    Approach (when unblocked): - Proxy importance (zero extra computation): use feature_drift_detected_total rate as importance signal — features that drift most affect model performance most. Available immediately once a model consumer is integrated. - Permutation importance: shuffle one feature column, measure prediction score drop. Requires a ModelProtocol interface and a labeled evaluation dataset. - SHAP: requires a tree or neural model with SHAP-compatible explainer. High compute cost.

    Acceptance Criteria (when unblocked): - [ ] A specific downstream model and evaluation dataset are identified. - [ ] Importance scores are persisted per (vector_id, feature_key, model_version). - [ ] Exposed via API endpoint or Grafana panel.

    Advanced Memory Telemetry — RSS vs mmap Split in AdaptiveConcurrencyController

    Blocked by: Spill system must have confirmed production usage before this distinction has any impact.

    Problem: AdaptiveConcurrencyController monitors total RAM pressure without distinguishing RSS (pages in RAM) from mmap (memory-mapped files via np.memmap in the spill layer). If a large batch spills to disk, RSS drops but mmap grows — the controller may incorrectly grant resources because it sees lower RAM pressure, while the system is actually I/O-bound on the mmap pages.

    Approach (when unblocked): Replace psutil.virtual_memory().percent with a two-signal model: - RSS: psutil.Process().memory_info().rss / psutil.virtual_memory().total - mmap: psutil.Process().memory_maps() summed for paths matching the spill directory

    Acceptance Criteria: - [ ] A spill-active production scenario is identified that triggers the mmap/RSS confusion. - [ ] AdaptiveConcurrencyController back-pressure threshold accounts for both signals independently. - [ ] No regression in controller behavior when spill is inactive (mmap signal is zero).

    Runtime Layer Extraction — bootstrap/, debug/catalog*, graph/, version promotion out of features/ into athomic/runtime/

    Trigger: Only pursue once a second real subsystem under athomic/ai/ (e.g. agents/, rag/) concretely needs the same artifact lifecycle (compile → persist → serve → promote/rollback) — not when it merely could reuse it in theory.

    Status (verified 2026-06-22): No such second consumer exists today. grep -rl "bootstrap|catalog|canary|promote|rollback|deployment" src/nala/athomic/ai/agents src/nala/athomic/ai/rag returns zero hits. features/bootstrap.py, features/debug/catalog_repository.py, features/graph/, and the version-promotion API (registry.promote()/rollback()) are all currently typed directly against VectorId/FeatureExecutionPlan — generalizing them now means designing a Protocol with no second real implementation to validate it against.

    Why not now: This is the same anti-pattern the project's Factory Rules already ban for *Factory classes ("only introduce a Registry when more than one real provider exists or is concretely planned") — extracting a generic runtime/ package for hypothetical AgentRuntime/PromptRuntime/RAGRuntime consumers is premature abstraction without a concrete second caller. The adaptive_concurrency precedent often cited for this move doesn't actually demonstrate proven reuse either — it has exactly one caller today (features/) too; it lives in a neutral namespace because admission/concurrency control is domain-agnostic by construction, not because two systems use it.

    Approach (when unblocked): - Extract narrow Protocols first (e.g. CompiledArtifactRepositoryProtocol, DeploymentControllerProtocol) without moving the implementation — shape the seam, don't build the wall. - Only move concrete code into athomic/runtime/ once the second consumer (Agents or RAG) has a real, shipped need for the same lifecycle, so the abstraction is validated against two real shapes instead of one imagined one. - Keep features/ owning the domain-specific parts (encoders, transformations, execution engines) regardless of what happens to the lifecycle plumbing.

    Acceptance Criteria (when unblocked): - [ ] A second subsystem (Agents, RAG, or other) has a concrete, shipped requirement for compile/persist/serve/promote/rollback lifecycle. - [ ] Protocols are extracted and validated against both real implementations before any code physically moves. - [ ] features/ make solid / import-linter contracts pass unchanged after extraction (no new circular imports introduced). - [ ] No behavior change for existing Feature Platform callers (FeaturePlatformService public API surface unchanged).

    Input Data Contracts — schema/domain validation at DataLoaderProtocol boundary

    Trigger: Only pursue once a real incident is observed where a malformed batch (corrupted schema, out-of-domain values) reaches ColumnarFeatureContext and causes a failure or silent bad output downstream — not preemptively.

    Status: Today validation is output-side only — DriftBaselineProtocol catches statistical anomalies after transformation. There is no input-side contract check at the DataLoaderProtocol boundary (io/ — ArrowAdapter, DictListAdapter) before a batch is admitted into the pipeline.

    Approach (when unblocked): - Add a narrow DataContractProtocol.validate(batch) -> ValidationResult checked at the loader boundary, before the batch reaches ColumnarFeatureContext. - Default/single-provider implementation: a lightweight in-house schema+domain check (column dtypes, declared value ranges) — no external dependency needed for the common case. - Only register a great_expectations-backed provider via the existing Strategy+Registry convention if the in-house check proves insufficient for a real failure mode (e.g. cross-column invariants) — don't pull in the dependency speculatively. - On violation: route the batch to a DLQ/reject path (mirroring the existing retry/dlq_builder.py pattern in messaging) rather than raising and failing the whole tier.

    Acceptance Criteria (when unblocked): - [ ] A specific incident (corrupted schema or out-of-domain values reaching ColumnarFeatureContext) is identified and documented. - [ ] Validation runs at the DataLoaderProtocol boundary, before tier execution, with zero added latency on the happy path. - [ ] Rejected batches are observable (Prometheus counter + structured log), not silently dropped. - [ ] No new external dependency added unless the in-house check is demonstrated insufficient.
