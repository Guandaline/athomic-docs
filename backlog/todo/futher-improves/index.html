    Title: Add Detailed Exception Logging in BaseService.close

    Description: The close method in BaseService currently catches generic Exceptions during the shutdown of its background task and the execution of the _close hook. This prevents a failing service from halting the entire application shutdown but can hide the root cause of the failure. The except blocks should be updated to use self.logger.exception(...) to log the full stack trace, which will significantly improve debuggability of shutdown-related issues.

    Affected Files: * nala/athomic/services/base.py

    Labels: improvement, observability, logging

    Acceptance Criteria: * The try...except block in the close method must log failures using logger.exception. * Tests should confirm that if a service's _close method raises an exception, it is logged with a full stack trace.

    Title: Add Documentation Regarding Dynamic Cache Decorator Application in HttpClientFactory

    Description: The HttpClientFactory dynamically applies the @cache decorator to the get method of a client instance when caching is enabled for that client. This is a powerful and efficient pattern but can be non-obvious to new maintainers. Add a clear and concise code comment in this section of the factory to explain why and how this is done. The comment should clarify that the get method on a cached client instance is a wrapped object, not the original method from the provider class.

    Affected Files: * nala/athomic/http/factory.py

    Labels: documentation, maintainability, low-priority

    Acceptance Criteria: * A descriptive comment is added to the if client_settings.cache... block in HttpClientFactory.create.

    • Title: Refactor Mongo Outbox Repository to Use Atomic Updates
    • Type: Refactor
    • Priority: High
    • Description: The current implementation in MongoOutboxRepository uses a find_one() followed by a save() for update operations like mark_event_attempted, mark_event_published, and mark_event_failed. This pattern is not atomic and can lead to race conditions under high load. It is also less performant than a single database operation.
    • Acceptance Criteria:
      • ```
      • [ ] The mark_event_attempted method should be refactored to use a single update_one operation with the $inc operator to increment the attempts field and $set to update last_attempt_at.
      • [ ] The mark_event_published method should use a single update_one operation with $set to change the event's status to PUBLISHED.
      • [ ] The mark_event_failed method should use a single update_one operation with $set to update the last_error and potentially the status fields.
      • [ ] The changes must ensure the operations remain atomic and performant.
      • ```
    • Files:
      • nala/athomic/database/outbox/mongo/mongo_outbox_repository.py
    • ID: BK-002
    • Title: Ensure Consistent Exception Propagation in BaseKVStore
    • Type: Bug
    • Priority: Medium
    • Description: The delete method in BaseKVStore catches exceptions, records them in the span, but does not re-raise them. This is inconsistent with the get and set methods, which correctly propagate exceptions. This inconsistency can lead to silent failures where callers are unaware that a delete operation failed.
    • Acceptance Criteria:
      • ```
      • [ ] Add a raise statement within the except block of the delete method in BaseKVStore.
      • [ ] Verify that all data operation methods (get, set, delete, exists, clear, etc.) in BaseKVStore have a consistent error handling strategy (i.e., they all propagate exceptions after logging/tracing).
      • ```
    • Files:
      • nala/athomic/database/kvstore/base.py
    • ID: BK-003
    • Title: Review Saga Compensation Strategy for Critical Failures
    • Type: Chore
    • Priority: Medium
    • Description: The current OrchestrationSagaExecutor implements a "fail-fast" strategy for compensations. If a compensation step fails, the saga is marked as FAILED, and the compensation chain is halted. For critical transactions, this could leave the system in an inconsistent state that requires manual intervention.
    • Acceptance Criteria:
      • ```
      • [ ] The team should discuss the implications of a failing compensation.
      • [ ] Decide if a retry mechanism should be added to compensation steps.
      • [ ] Consider implementing an alerting mechanism (e.g., publishing an event to a "dead-letter-saga" topic) when a compensation fails permanently.
      • [ ] Document the chosen strategy and its trade-offs.
      • ```
    • Files:
      • nala/athomic/resilience/sagas/executors/orchestration_executor.py
    • ID: BK-004
    • Title: Improve Security in RedisKVClient Connection Logging
    • Type: Refactor
    • Priority: Low
    • Description: The _connect method in RedisKVClient logs the connection URI with a note that credentials have been redacted. While the intention is good, logging the full URI string, even with a note, poses a potential risk of leaking credentials if the redaction process fails or is incomplete. A more secure approach is to never include sensitive parts of the URI in logs in the first place.
    • Acceptance Criteria:
      • ```
      • [ ] The log message in the _connect method should be modified to explicitly exclude the user/password part of the URI.
      • [ ] The log should still clearly indicate the host, port, and database being connected to for debugging purposes.
      • ```
    • Files:
      • nala/athomic/database/kvstore/providers/redis/client.py
