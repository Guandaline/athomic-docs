    TASK: Middleware HTTP/ASGI de replay idempotente

    Data: 29 de Junho de 2026 Autor: Nala Assistance Status: Proposta (não-bloqueante) Origem: verificação durante o M1 do Nala Ori (idempotência HTTP via Idempotency-Key)

    1. Contexto

    O Nala Ori (consumidor do athomic) implementa a convenção de wire Idempotency-Key no seu boundary HTTP: um POST repetido com a mesma chave deve devolver byte-a-byte a resposta original (status, headers, body), sem repetir efeitos colaterais. No M1 o Ori resolve isso com um store Postgres próprio (nala.ori.core.wire.idempotency).

    A verificação confirmou que o athomic já entrega todo o core de idempotência, e é Redis-capable:

    • nala.athomic.resilience.idempotency.KVIdempotencyStorage — store sobre KVStoreProtocol (providers redis e local existem), com wrap em IdempotencyRecord (distingue "resultado None" de "cache miss") e set(nx=True).
    • IdempotencyHandler — fluxo Cache-Aside + Single-Flight com lock distribuído (LockingFactory), além do modo batch para consumers (acquire_batch/confirm_batch/release_batch).
    • @idempotent (decorator), IdempotencySettings (enabled, ttl_seconds, lock_timeout_seconds, kvstore) e o model IdempotencyResponse (status_code, headers, body) — pensado justamente para guardar uma resposta HTTP.

    Lacuna identificada: não existe no athomic um adaptador HTTP/ASGI que capture a resposta de um handler (status/headers/body), a persista via IdempotencyResponse keyada pelo header Idempotency-Key, e a reproduza em replays — incluindo o single-flight (concorrência na mesma key → 409/aguarda) e a validação de fingerprint do request (mesma key + body diferente → 422).

    Hoje cada serviço HTTP (Ori e futuros) reescreveria essa cola. Como o athomic é o substrato e a filosofia é "ter a capacidade pronta antes do consumidor precisar", o replay HTTP idempotente deve morar aqui, reutilizando o core acima.

    Não-bloqueante para o Ori: o Ori consegue migrar do store Postgres para a idempotência do athomic apenas trocando o backend para KVIdempotencyStorage + IdempotencyHandler, mesmo antes deste middleware existir. Esta tarefa elimina a cola HTTP duplicada; não é pré-requisito do M2 do Ori.

    2. Objetivo

    Entregar um componente HTTP-agnóstico (camada integration/web do athomic) que torne um endpoint idempotente de ponta a ponta a partir do header Idempotency-Key, sobre o core de resilience.idempotency já existente.

    3. Backlog de Implementação

    Épico: EPIC-IDEM-HTTP — Replay idempotente no boundary HTTP

    • [x] TASK-IDEM-001: Definir o contrato do envelope HTTP persistido reusando IdempotencyResponse (status_code, headers serializáveis, body bytes) + fingerprint do request (hash de método+path+body) para detectar key-reuse com payload divergente.
    • [x] TASK-IDEM-002: Implementar um middleware/dependency ASGI (FastAPI-first, mas sem acoplar o core a FastAPI) que: extrai Idempotency-Key; resolve a key contextual (tenant-aware via ContextKeyGenerator); consulta KVIdempotencyStorage; em hit, replica a resposta; em miss, executa sob single-flight (IdempotencyHandler/lock) e persiste o envelope.
    • [x] TASK-IDEM-003: Semântica de concorrência: requests simultâneos na mesma key → segundo recebe 409 Conflict (ou aguarda até lock_timeout, configurável) em vez de duplicar efeito.
    • [x] TASK-IDEM-004: Validação de reuso: mesma key + fingerprint diferente → 422 com Problem Details (code: idempotency_key_reuse).
    • [x] TASK-IDEM-005: Settings: estender/usar IdempotencySettings para o escopo HTTP (status de replay, headers a preservar/excluir — ex.: Date).
    • [x] TASK-IDEM-006: Observabilidade: métricas idempotency_hit/miss/conflict e logs estruturados com a key resolvida.
    • [x] TASK-IDEM-007: Testes de integração com KV redis e local: replay exato, single-flight sob concorrência, key-reuse divergente, expiração TTL.

    Próximos passos / verificação

    4. Implementation status (delivered)

    TASK-IDEM-001..007 are implemented in src/nala/athomic/http/idempotency/. Key facts and deviations from the original proposal (all deliberate):

    • Home is athomic/http/idempotency/, not integration/web. There is no integration/web layer; athomic/http/ already exists (outbound client). The server-side capability now lives beside it: models.py (envelope + compute_request_fingerprint), processor.py (HttpIdempotencyProcessor, framework-agnostic), asgi.py (HttpIdempotencyMiddleware, pure ASGI — no FastAPI import). Wired in api/startup/setup.py before RequestContextMiddleware so the tenant context is populated before key resolution.
    • DRY single-flight. The Cache-Aside + Single-Flight algorithm was extracted into IdempotencyHandler.run_single_flight(storage_key, compute, *, ttl, validate_existing, should_store); both the @idempotent decorator (execute) and the HTTP processor now share it. validate_existing enforces the request fingerprint (→ 422 on reuse); should_store skips caching transient >=500 responses so clients can retry.
    • Envelope does NOT reuse IdempotencyResponse verbatim — the KV serializer JSON-encodes stored values and rejects raw bytes, and byte-exact replay must preserve header order/duplicates. So StoredHttpResponse persists the body base64-encoded and headers as ordered (name, value) pairs (volatile headers like Date/Server stripped; uvicorn re-adds them). Rationale documented in models.py.
    • Observability: a single labelled counter idempotency_http_requests_total{outcome,http_method} (outcome ∈ hit/miss/conflict/reuse_conflict/bypass) instead of four separate counters — same information, idiomatic Prometheus.
    • Settings: IdempotencySettings.http (IdempotencyHttpSettings) with enabled, key_header, enabled_methods, wait_for_result, excluded_headers.
    • KV connection: the processor lazily connects its KV client on first request (lock-guarded), because Redis does not connect lazily and would otherwise silently no-op.
    • Security (from the Guardian security review): (1) auth/session response headers (set-cookie, www-authenticate, authorization) are stripped from the stored envelope by default, so a replay can never leak another caller's session; (2) the storage key is scoped by a SHA-256 caller_identity derived from the configured identity_headers (default Authorization) in the ASGI middleware, so two different callers can never share a replay slot on the same client-chosen key — independent of request-context/tenant population. Unauthenticated requests fall back to safe key-only scoping; cookie/API-key deployments add their auth header to identity_headers.
    • Tests: tests/unit/nala/athomic/http/idempotency/ (models, processor, ASGI round-trip) + tests/unit/nala/athomic/resilience/idempotency/test_unit_run_single_flight.py (extracted-primitive regression) + tests/integration/nala/athomic/http/idempotency/ (parametrized over local and redis KV; redis needs make infra-up).

    5. Follow-ups (post-merge, tracked separately)

    • [ ] TASK-IDEM-009 (max-body cap): eligible POST/PATCH requests buffer the full request body and full response body in memory (and base64 the response for storage). Bounded and fine for JSON APIs, but there is no configurable size cap — add a max_body_bytes setting that bypasses idempotency (or streams through untouched) above the threshold, so a large upload/response under an Idempotency-Key cannot be fully materialized. Raised by the Guardian performance judge (info-level).
    • See TASK-auth-context-population-ordering.md for the broader auth/context timing fix that would let tenant/user scoping add defense-in-depth on top of the caller_identity hash (TASK-IDEM-scoping hardening depends on it).
    • See TASK-importlinter-http-layer.md and TASK-guardian-dossiers-idempotency.md.
