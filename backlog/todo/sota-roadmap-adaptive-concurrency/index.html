    SOTA Roadmap — Adaptive Concurrency Controller

    Status: LIVING DOC · Owner: resilience platform · Created: 2026-06-21

    0. Purpose

    The goal is to claim "state-of-the-art" for src/nala/athomic/resilience/adaptive_concurrency/ only when it is backed by measured evidence, not feature count. This document is the map: an honest capability gap analysis versus the reference systems, the original contributions that already set us apart, a value-ranked sequence of remaining epics — including long-horizon items, since time is explicitly not a constraint for this roadmap — and an objective definition of "SOTA-ready" so the claim is data-driven. It is a living doc — update statuses as epics land.

    Reference systems: Netflix Concurrency Limits / Gradient2, Envoy Adaptive Concurrency, Mesos DRF, Kubernetes API Priority and Fairness (APF). These are the same baselines named in the research dossier (docs/backlog/research/distributed-adaptive-resource-scheduler/), which this roadmap feeds: the dossier is the scientific positioning, this document is the engineering path that proves or disproves it.

    Related, non-duplicated docs: - adaptive_concurrency_backlog-v2.md (DONE) — the 8 features that took the controller from a plain AIMD limiter to its current shape (queue TTL, targeted wakeup, OTel tracing, memory double-counting fix, weighted pressure, QoS/priority, DRF fairness, cost learning). - adaptive-concurrency-hardening.md (TODO) — 3 small correctness/structure fixes (composition-root extraction, active-probe pressure gate, post-startup cost-profile convergence). Those are hardening, not SOTA gaps; this document does not repeat them. - architecture-scorecard.md (DELIVERED) — the objective module-scoring tool (scripts/archscore/). Reused as-is for §4, not reinvented here.

    1. Where we already lead (original contributions)

    These are not parity items — they are things most/all reference systems do not do:

    1. Pluggable control law behind a single protocol — AIMD, Netflix-style dual-window Gradient2, and SLO-driven proportional control all implement ConcurrencyAlgorithmProtocol and are hot-swappable via settings (algorithms/factory.py). Reference systems each ship exactly one control law; we ship the breadth of three under one interface.
    2. Single-resource Dominant Resource Fairness in an application-level admission controller — DRF (cost + memory dimension + per-class weights) is otherwise essentially confined to cluster schedulers (Mesos, YARN, Borg); we have not found another in-process middleware library that implements it (fairness/drf.py:96-126).
    3. Atomic, self-healing distributed admission budget — the reservation is admitted atomically inside the KV store (no Python-side check-then-write race) and expires via TTL if an instance crashes mid-hold (distributed/budget.py:43-60).
    4. Sample-count-weighted distributed cost-profile merge — each instance owns its own hash field; load_profiles merges every non-stale instance weighted by total_executions, so a cold replica cannot clobber a warm one's baseline (distributed/cost_store.py:84-115).
    5. Targeted-wakeup priority queue with O(1) starvation promotion — no notify_all thundering herd; lazy deletion keeps len() O(1) even under heavy churn; a starved ticket is promoted by re-pushing a linked ticket sharing the same asyncio.Event, not by rebuilding the heap (admission/queue.py:202-237).

    3. Value-ranked epics (the sequence)

    Ranked by SOTA value per effort. Per the "time is not a constraint, always future-ready" philosophy, every horizon is listed — but long-horizon items carry an explicit evidence trigger instead of being scheduled blind.

    Tier 1 — unlocks every other claim in this document

    1. Comparative benchmark harness. Synthetic heavy-tail workloads first (e.g. 80/15/5 short/medium/expensive request mix), then one realistic AI-serving workload. Baselines: Fixed Semaphore, plain AIMD, plain Gradient2 — chosen because they're comparable in-process mechanisms, unlike Mesos/Ray which operate at the cluster layer (see §5). ≥30 runs per configuration, confidence intervals reported. This is the prerequisite for every other row in §2 to become a real claim rather than a feature checkbox.
    2. Composed-pipeline stability characterization, using the Tier-1 harness: measure overshoot, settling time, and oscillation under burst + memory-pressure scenarios for the full stack (algorithm → fleet damper → active prober), not each mechanism in isolation.

    Tier 2 — fixes the predictive signal's foundation

    1. Per-task memory attribution. Replace the host-level RAM delta (controller.py:709-715) with tracemalloc-scoped or cgroup-delta accounting. The cost learner is the most "predictive" piece of the system and is currently trained on a signal contaminated by concurrent neighbors — this is the highest-leverage correctness fix for SOTA credibility.

    Tier 3 — fairness and coordination reach the distributed level the name implies

    1. Fleet-wide DRF. Extend DominantResourceFairnessTracker beyond a single instance, reusing the already-wired SharedFleetPressureStore transport rather than building new plumbing.
    2. Bidirectional fleet coordination. Today FleetCooperativeDamper only ever reduces capacity; investigate a cooperative-increase signal (with a documented stability argument) so the fleet can also expand together, not just contract together.

    Tier 4 — QoS and control loop closure

    1. Priority-aware distributed budget — today fleet-wide admission is priority-blind; a low-priority reservation in instance B can exhaust budget that a high-priority request needs in instance A.
    2. Per-workload-class SLO control, with anti-windup investigated before any integral term is added — measure residual steady-state error and settling time via the Tier-1 harness first; do not assume PID is required without that evidence (the current proportional-only design may already be the right choice).

    Tier 5 — long horizon, evidence-gated (the "always future-ready" tier)

    Each item here is deliberately speculative and must not be started without its trigger firing — this keeps the roadmap future-proofed without encouraging premature complexity.

    1. Meta-learning of PressureComposer weights / Gradient2 tolerances (e.g. Bayesian optimization closing the last manually-tuned loop). Trigger: Tier 1 harness shows static weights measurably underperform across ≥2 distinct workload shapes.
    2. Predictive/forecasting admission control — anticipate congestion before saturation rather than reacting to it. Trigger: Tier 1/2 data shows reactive control's lag (not its target value) is the dominant source of SLO violations.
    3. Hierarchical scheduling / quota trees for large multi-tenant deployments. Trigger: a concrete multi-tenant deployment need arises; do not build ahead of a real consumer.

    Evaluated and parked

    • Placement/topology-aware scheduling — out of category. This module is in-process admission control, not a cluster scheduler; comparing it against Ray/Volcano/Mesos on this axis mixes operational layers (the same baseline-comparability risk flagged in the research dossier's own threats section). Not a gap to close — a different product.
    • RL-assisted scheduling — parked behind the Tier 5 evidence triggers. Adding a trainable policy before classical control is shown to be insufficient would be complexity without a justified need.

    4. Definition of "SOTA-ready" (objective claim gate)

    Claim SOTA when all hold (measured, not judged):

    1. Comparative evidence: benchmark executed against ≥3 baselines under ≥2 workload classes, with a measured improvement (not just feature presence), n≥30 per configuration, confidence intervals reported.
    2. Composed-pipeline stability: no oscillation/instability within the documented operating envelope; overshoot and settling time reported with numbers, not described qualitatively.
    3. Memory-attribution accuracy: learned ewma_bytes validated against an isolated single-task ground truth (no concurrent neighbors) — error bound reported.
    4. Architecture scorecard (scripts/archscore/, already DELIVERED) ≥ target for this module's role profile.
    5. Quality gate green repo-wide (already enforced: mypy · lint · tests).
    6. If the goal is publication: a public reproducibility artifact (harness + datasets + figures), linked from the research dossier.

    5. Non-goals

    • Becoming a cluster scheduler (placement, topology, gang scheduling) — a different operational layer than this module occupies.
    • RL-assisted scheduling without its Tier 5 evidence trigger firing.
    • A hosted UI or multi-tenant SaaS control plane — out of scope for an in-process admission controller.
