    Standardize all *Factory classes on Factory+Registry (project-wide audit)

    Status: TODO — not started · Owner: platform/quality · Created: 2026-06-22

    Context

    CLAUDE.md documents Factory + Registry as the project's standard pattern for configurable provider creation ("New providers are registered without modifying core code"), with KVStoreFactory, SecretsFactory, ProducerFactory and EngineFactory cited as canonical examples.

    A full project-wide audit (grep -rn "^class.*Factory" src/nala/) found 122 *Factory classes. Cross-checking each against the documented pattern shows the convention is followed inconsistently:

    Category Count Definition
    Completa 39 Registry-backed, config via constructor parameter, no inconsistent singleton
    Parcial 35 Has a Registry but deviates (internal get_settings(), singleton, or both)
    Trivial/fora do padrão 48 No Registry at all — either a single-method passthrough (return SomeClass(**kwargs)) or selection logic implemented ad hoc instead of via Registry

    This is now also an objective, ongoing signal: make archscore scores every file with a *Factory class on a new factory_pattern dimension (see architecture-scorecard.md's 2026-06-22 entry) — penalizing missing Registry (-40), internal get_settings() (-30), and single-method triviality (-30). This backlog is the remediation plan for what that dimension already measures.

    Parcial (35) — has Registry, but still deviates

    These already follow the structural shape (Registry-backed provider selection) but mix in a Service Locator and/or an inconsistent singleton. Lower priority than the trivial group — fixing them is a smaller, mechanical diff (remove get_settings() call, inject the settings object instead).

    Representative sample (full list reproducible via the audit query below): AgentFactory, CheckpointFactory, CognitiveFactory, TokenServiceFactory, FeaturePlatformFactory, DocumentRepositoryFactory, GraphDatabaseFactory, OutboxStorageFactory, VectorStoreFactory, EventSourcingFactory, EventBusFactory, ExecutionProviderFactory, HttpClientFactory, SettingsProviderFactory, ServiceDiscoveryFactory, ProducerFactory, DLQHandlerFactory, DeadLetterStrategyFactory, OutboxExecutorFactory, OutboxPublisherFactory, OutboxRouterFactory, TaskBrokerFactory, LineageStoreFactory, NotificationFactory, PayloadProcessorFactory, CompressionMiddlewareFactory, LockingFactory, RateLimiterFactory, SmartRouterFactory, SecretsFactory, SerializerFactory, StorageFactory, UsageEmitterFactory, WorkflowClientFactory, SchedulerFactory.

    Tasks

    1. Triage the 48 "trivial/fora do padrão" factories into three buckets:
    2. (a) Genuinely needs Registry — more than one real provider implementation exists or is plausible soon (e.g. GuardFactory, GraphStoreFactory, ReRankerFactory, RetryFactory, CircuitBreakerFactory). Add a *Registry (Strategy+Registry, same shape as TransformationRegistry) and route create() through it.
    3. (b) Legitimately single-provider today — only one concrete implementation exists and none is planned (verify via actual callers, not assumption). For these, do not force a Registry — per [[feedback-configurability-principle]], don't add dead config for a provider axis that doesn't exist. Document the decision inline (one-line comment: "single provider by design, see factories-pattern-standardization.md") so the next reviewer doesn't re-flag it.

    4. (c) Base/protocol factories — FactoryProtocol (base/factory.py) and similar are intentionally minimal (they're the interface, not an implementation) — exclude from remediation, see "Not in scope".

    5. Remove get_settings() Service Locator calls from every factory in both groups (trivial + parcial) that has one — inject the needed settings object via the create()/__init__ parameter instead. This is the same root issue as Task 4 in messaging-module-architecture-cleanup.md, scoped here to the rest of the codebase. Do this after Task 1's Registry decisions land for a given factory, to avoid touching the same file twice.

    6. Resolve singleton inconsistency case by case. For each factory with a cached _instance, decide: is "one instance per process" a real constraint (e.g. a connection pool that must not be duplicated) or an accidental carry-over? If real, document why in the class docstring. If accidental, remove the cache — ProducerManagerFactory (messaging/producers/manager_factory.py) is the project's own precedent for "factories are not singletons by default."

    7. ReadStrategyFactory / WriteStrategyFactory (database/kvstore/fallback/) are large files (310 / 185 lines) with no Registry — investigate whether their size comes from inlined strategy-selection logic that a Registry would directly replace, or from unrelated logic that happens to share the file. Likely the single best ROI item in this backlog (big file, clear Registry shape, no get_settings() complication).

    8. Re-run make archscore after each batch of fixes and compare factory_pattern breakdown deltas — this is the dimension's primary use case, so use it to confirm progress instead of re-doing the manual audit.

    Suggested order

    1. Task 4 (ReadStrategyFactory/WriteStrategyFactory) — highest ROI, isolated.
    2. Task 1 triage for the remaining 47 trivial factories — produces the actual work list (some will turn out to be bucket (b), out of scope).
    3. Task 1(a) Registry introductions for whatever triage flags as bucket (a).
    4. Task 3 (singleton case-by-case) — can run in parallel with Task 1(a), it touches different lines in the same files.
    5. Task 2 (get_settings() removal) — do last across both trivial and parcial groups, since Tasks 1/3 may change the constructor shape these calls live in.

    Acceptance

    • make archscore mean for factory_pattern improves measurably for every file touched (no regression tolerance needed beyond the existing --tolerance 2.0 gate).
    • make test and make mypy green.
    • Every bucket-(b) decision ("single provider by design") has a one-line comment in the file, not just a note in this backlog — so archscore's flag doesn't get re-litigated by the next person who runs the audit.
    • No behavioral change — this is structural (Registry introduction + DI) not a feature change.

    Not in scope

    • athomic/base/factory.py (FactoryProtocol) — this is the Protocol definition itself, not a concrete factory; a single abstract method is correct here by definition, not a violation.
    • athomic/integration/messaging/producers/manager_factory.py (ProducerManagerFactory) — already correctly non-singleton; flagged only for its single-method shape, which is appropriate for a thin factory that has nothing to select between (it always builds the same ProducerManager shape from settings). Re-evaluate only if a second producer-manager variant appears.
    • The full MessagingManagerFactory rework — already tracked with more context in messaging-module-architecture-cleanup.md Task 3; this backlog just lists it for completeness in the project-wide count.
    • Adding new scorer dimensions beyond factory_pattern (e.g. detecting inconsistent singleton automatically) — singleton-vs-not is too contextual to score objectively without false positives; Task 3 handles it manually.

    Effort

    Large (1–2 weeks) — 48 trivial + 35 partial factories across ~40 subsystems; mechanical per-file but touches a meaningful fraction of athomic/'s composition roots. Sequence behind messaging-module-architecture-cleanup.md and admission-control-module-extraction.md since those are smaller, already-scoped efforts in adjacent code.
