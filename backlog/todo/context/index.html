    EPIC: Eliminate Global Service State

    Overview

    The Athomic framework currently relies on a small number of globally accessible singleton instances and factory-managed cached objects to provide infrastructure services across the application lifecycle.

    While this approach simplifies access to shared services, it introduces hidden dependencies, complicates test isolation, reduces runtime flexibility, and creates architectural constraints for future scenarios such as multi-tenancy, isolated workers, embedded runtimes, plugin sandboxing, and advanced dependency injection strategies.

    This epic aims to eliminate global service state while preserving Athomic's architectural principles:

    • Explicit over implicit
    • Constructor Injection over Service Locator
    • Composition Root over distributed initialization
    • Predictable lifecycle management
    • Minimal runtime magic
    • High testability

    The final architecture should ensure that all infrastructure services are owned and managed explicitly by the Athomic composition root, without requiring global variables, hidden singleton accessors, or runtime service discovery.

    Architectural Goals

    Primary Goals

    • Remove all framework-level global service state.
    • Eliminate hidden dependencies.
    • Improve test isolation.
    • Improve lifecycle ownership.
    • Improve runtime flexibility.
    • Preserve explicit dependency flow.

    Non-Goals

    This epic does not introduce:

    • Automatic dependency injection.
    • Reflection-based service resolution.
    • Service Locator pattern.
    • Auto-wiring.
    • Annotation-based dependency management.
    • Third-party DI containers.

    Athomic should remain fully explicit regarding dependency creation and ownership.

    Current Problems

    Global Athomic Instance

    Current implementation:

    Path: src/nala/athomic/provider.py

    _athomic_instance: Optional[Athomic]

    Issues:

    • Hidden dependency source.
    • Difficult test isolation.
    • Global mutable state.
    • Tight coupling to application lifecycle.
    • Prevents multiple Athomic runtimes in the same process.

    Factory-Managed Singleton State

    Current examples:

    Path: src/nala/athomic/database/factory.py

    connection_manager_factory._instance

    Potentially similar patterns across:

    • Database
    • Messaging
    • AI
    • Security
    • Lifecycle

    Issues:

    • Runtime state exists outside the composition root.
    • Service ownership becomes unclear.
    • Startup/shutdown responsibilities become fragmented.
    • Hidden caching behavior.

    Dependency Discovery Instead of Dependency Ownership

    Current usage pattern:

    connection_manager_factory.create()

    Instead of:

    athomic.connection_manager

    This creates ambiguity regarding:

    • Who owns the instance.
    • Who initializes it.
    • Who shuts it down.
    • Whether it is safe to reuse.

    Target Architecture

    Single Composition Root Ownership

    All infrastructure services must be created and owned by Athomic.

    Example:

    class Athomic:

    def __init__(self):
        self.connection_manager = ...
        self.secrets_manager = ...
        self.plugin_manager = ...
        self.lifecycle_manager = ...

    The Athomic instance becomes the authoritative runtime context.

    Explicit Dependency Flow

    Dependencies should be passed through constructors.

    Preferred:

    class UserRepository:

    def __init__(
        self,
        connection_manager: ConnectionManager,
    ):
        self.connection_manager = connection_manager

    Avoid:

    connection_manager_factory.create()

    Avoid:

    get_athomic_instance()

    Runtime Context Ownership

    All runtime services should be reachable from the Athomic runtime object.

    Example:

    athomic.connection_manager

athomic.secrets_manager

athomic.plugin_manager

athomic.lifecycle_manager

    No global lookup should be required.

    Workstream 1: Remove Global Athomic Instance

    Objective

    Eliminate framework-wide access through global variables.

    Task 1.1

    Deprecate:

    Path: src/nala/athomic/provider.py

    get_athomic_instance()

    Success Criteria:

    • Mark as deprecated.
    • Emit warning when used.
    • Document replacement path.

    Task 1.2

    Refactor internal consumers.

    Replace:

    get_athomic_instance()

    With explicit dependency injection.

    Success Criteria:

    • No internal framework code depends on global Athomic state.

    Task 1.3

    Remove global provider.

    Success Criteria:

    • _athomic_instance no longer exists.
    • Framework startup remains functional.

    Workstream 2: Eliminate Factory-Owned Runtime State

    Objective

    Move runtime ownership from factories to Athomic.

    Task 2.1

    Audit all factories.

    Identify:

    • Cached instances.
    • Singleton implementations.
    • Runtime state storage.

    Deliverable:

    Factory Inventory Report

    Task 2.2

    Refactor ConnectionManager ownership.

    Current:

    connection_manager_factory._instance

    Target:

    athomic.connection_manager

    Success Criteria:

    • Factory becomes stateless.
    • Instance ownership moves to Athomic.

    Task 2.3

    Apply same pattern to remaining services.

    Candidates:

    • Messaging
    • Secrets
    • AI managers
    • Plugin managers
    • Lifecycle services

    Success Criteria:

    • No factory stores runtime instances.

    Workstream 3: Introduce Runtime Context Model

    Objective

    Formalize ownership of infrastructure services.

    Task 3.1

    Create runtime context abstraction.

    Path:

    src/nala/athomic/runtime/

    Proposed structure:

    runtime/
├── __init__.py
├── context.py
└── protocol.py

    Task 3.2

    Implement AthomicRuntimeContext.

    Example:

    class AthomicRuntimeContext:
    settings: AppSettings
    connection_manager: ConnectionManager
    secrets_manager: SecretsManager
    plugin_manager: PluginManager
    lifecycle_manager: LifecycleManager

    Purpose:

    • Explicit runtime ownership.
    • Centralized service access.
    • Improved observability.

    Task 3.3

    Expose runtime context through Athomic.

    Example:

    athomic.runtime.connection_manager

    Success Criteria:

    • All infrastructure services reachable through runtime context.

    Workstream 4: Improve Test Isolation

    Objective

    Allow multiple isolated framework runtimes.

    Task 4.1

    Remove global cleanup requirements.

    Current:

    clear_athomic_instance()

connection_manager_factory.clear()

    Target:

    athomic = create_test_runtime()

    Success Criteria:

    • No global cleanup required between tests.

    Task 4.2

    Enable parallel runtime execution.

    Support:

    athomic_a

athomic_b

athomic_c

    within the same process.

    Success Criteria:

    • No shared mutable framework state.

    Workstream 5: Strengthen Lifecycle Ownership

    Objective

    Ensure startup and shutdown ownership are centralized.

    Task 5.1

    Verify all services are created during bootstrap.

    Success Criteria:

    • No lazy global initialization.
    • No hidden runtime creation.

    Task 5.2

    Verify all services are shutdown through lifecycle manager.

    Success Criteria:

    • No orphaned resources.
    • Deterministic shutdown sequence.

    Migration Strategy

    Phase 1

    Introduce runtime context.

    Risk: Low

    Phase 2

    Migrate internal services.

    Risk: Medium

    Phase 3

    Deprecate global providers.

    Risk: Medium

    Phase 4

    Remove global providers.

    Risk: High

    Requires major-version release.

    Acceptance Criteria

    • No framework-level global mutable state.
    • No _athomic_instance.
    • No factory-owned runtime instances.
    • All infrastructure services owned by Athomic runtime.
    • All dependencies injected explicitly.
    • Multiple Athomic runtimes can coexist in the same process.
    • Parallel test execution supported.
    • No Service Locator pattern introduced.
    • No auto-wiring introduced.
    • No reflection-based dependency resolution introduced.
    • Startup and shutdown remain deterministic.
    • Existing architectural principles remain intact.

    Expected Benefits

    Architecture

    • Clear ownership boundaries.
    • Reduced coupling.
    • Increased maintainability.

    Testing

    • True runtime isolation.
    • Parallel execution support.
    • Simpler fixture management.

    Operations

    • Predictable lifecycle behavior.
    • Easier debugging.
    • Better observability.

    Future Evolution

    • Multi-tenancy readiness.
    • Embedded runtime support.
    • Worker isolation.
    • Plugin sandboxing.
    • Safer framework extensibility.

    Estimated Complexity

    Medium-High

    Most changes are architectural refactors rather than new feature development.

    Estimated effort:

    • Design: 1-2 days
    • Refactoring: 3-5 days
    • Testing & migration: 2-3 days

    Total:

    6-10 engineering days
