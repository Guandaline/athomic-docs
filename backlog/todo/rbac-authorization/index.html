    RBAC / Authorization — security/auth

    Status: STAGE 1 + 2 (seam) + KV resolver DONE · Owner: security/auth · Created: 2026-06-21

    Update 2026-06-22 — §6.4 DB/provider resolver seam made real + verified. The "dynamic-org resolver, on demand" item: the registry seam was confirmed capable (BaseRegistry.create(name, settings=…, **kwargs) injects dependencies), and a working KV-backed resolver now ships: - resolvers/kv.py::KVPermissionResolver — role→permissions sourced from a KV store, seeded from AuthorizationSettings.roles (usable before first refresh / when the key is absent). Authorization checks are hot-path, so permissions_for stays sync (reads an in-process cache); an async refresh() reloads from KV (KV authoritative when present; corrupt payload keeps the last good cache). Same cache + async-refresh pattern as release consensus — a naive per-call async lookup is deliberately avoided. - Registered under "kv"; build_authorization_service(settings, kv_store=…) forwards the client and raises a clear ValueError if a resolver needs a kv_store that was not provided. StaticPermissionResolver tolerates the uniformly-injected dep it does not use (**_). - Left to the deployment (the genuine "on demand"): when to refresh() (once at the composition root, and/or a periodic loop — mirroring the opt-in release-refresh loop). The resolver + seam are ready; no background wiring is forced on single-tenant/static deployments. 8 unit tests.

    Update 2026-06-22 — Stage 1 (§6.1) shipped. The modern replacement for require_role (kept for back-compat). - get_authorization_service() — lru-cached provider built from security.auth.authorization settings (defaults to dry_run when unconfigured), so enabling RBAC is a pure settings change. - 5 end-to-end tests through FastAPI with a real service (test_rbac_requires_with_routes.py): enforce allow/deny-403, scope grant, dry_run permissive, disabled permissive.

    0. Scope ownership — enforcement is at the API boundary, not in domains

    RBAC is a security/auth responsibility, not a feature-platform one. By the time a call reaches features (or any domain service), the request has already been authenticated and authorized at the API edge. No domain module imports authz, and features needs zero auth awareness.

    HTTP request
  → API route: Depends(require_auth(...))   # authentication
              Depends(requires("features:promote"))  # authorization (RBAC)
  → service.releases.promote_version(...)   # features: already-authorized call

    The features module only declares which permission an operation needs (as route metadata in the API layer, §3) — security/auth enforces. The one thing that may cross the boundary is identity-as-data (tenant_id for data scoping, sub for audit), passed as a parameter — never authz logic, and only where a concrete need exists. This doc plans the security-module work.

    1. Honest gap analysis (what exists today)

    More exists than "RBAC missing" — the foundations are there, the enforcement layer is not:

    Piece State
    Authentication (API key / JWT / IP) ✅ auth/providers/*, auth/executors/*
    Principal carries identity ✅ AuthenticatedUser has sub, roles, scopes, jti, exp
    Multi-tenancy id ✅ APIClient.tenant_id
    Route auth-policy selection ✅ AuthService.authorize(ctx, policy_name) → executor (public/private/internal)
    Coarse role gate ⚠️ require_role("admin") API dep — single-string role in user.roles, scattered in routes
    Permission model (resource:action) ❌
    Role → permission mapping ❌
    Scope enforcement (scopes are carried, never checked) ❌
    Central authorization service (allow/deny + audit) ❌
    Tenant-aware authz (tenant_id exists, unused in checks) ❌
    Resource-scoped permissions (promote vector X not Y) ❌ (advanced)

    So: authentication is solid; authorization is a coarse, ad-hoc role check. RBAC = turn that into a real permission model with a central enforcement point.

    2. Target model

    2.1 Permission

    A Permission is a string "<resource>:<action>" (conventional, greppable): e.g. features:promote, features:train, features:read, secrets:read. (Wildcards features:* / *:* allowed for admin roles.)

    2.2 Principal → permissions

    A principal is authorized for a permission P when:

    P ∈ direct_scopes(principal)  ∪  permissions(roles(principal))
    • scopes are treated as direct permissions (OAuth-style) — already on the principal, currently ignored.
    • roles expand to permissions via a role→permission mapping (the new bit).

    2.3 Role → permission mapping (pluggable — configurability principle)

    PermissionResolverProtocol.permissions_for(role) -> set[str]: - StaticPermissionResolver (default) — mapping from settings/TOML, reviewed as code. e.g. admin = ["*:*"], ml_engineer = ["features:*"], analyst = ["features:read", "features:train"]. - ProviderPermissionResolver (seam) — DB/Vault-backed for dynamic orgs. Registry-selected; default static. Not built until a dynamic need exists.

    2.4 AuthorizationService (the central enforcement point)

    class AuthorizationService:
    def is_allowed(self, principal: AuthenticatedUser, permission: str,
                   tenant_id: Optional[UUID] = None) -> bool: ...
    def require(self, principal, permission, tenant_id=None) -> None:  # raises
    • Wildcard match, role expansion via the resolver, tenant check (principal's tenant must match the resource tenant when provided).
    • Emits authz_decisions_total{permission, result} + structured deny logs (audit).
    • Raises AuthorizationError (new, in security/auth/exceptions.py) on deny → API maps to HTTP 403.

    2.5 Enforcement seam (API layer)

    A FastAPI dependency replacing scattered role strings:

    @router.post(..., dependencies=[Depends(requires("features:promote"))])

    requires(permission) resolves the current principal (existing require_token) and calls AuthorizationService.require(...). Keep require_role working (back-compat) but document requires(permission) as the path forward.

    3. How feature-platform operations get governed (without touching features)

    The sensitive control/release operations map to permissions, declared at the API route, not in the features module:

    Operation Permission
    serving.encode* / get_online_features features:read
    control.get_historical_features features:train
    releases.promote_version / rollback_version / set_canary features:promote
    control.discovery.* features:read

    The features module stays unaware of authz; only the API routes that expose these add Depends(requires("features:...")). This preserves the clean module boundary.

    4. Configurability (per the principle)

    • Role→permission mapping is committed config (static resolver default); a provider-backed resolver is a pluggable seam (registry), not built until needed.
    • Wildcards configurable; permission strings are data, not code.
    • enabled master switch — when off, authz is permissive (back-compat for deployments not yet using RBAC), with a loud startup warning.

    5. Failure modes / risks

    Risk Guard
    Lock everyone out on rollout enabled=false default + permissive mode + audit-only "dry-run" mode that logs would-be denials without enforcing
    Tenant data crossing tenant check in is_allowed when resource tenant supplied
    Permission sprawl conventioned resource:action + wildcard roles + a registry of known permissions
    Scope/role confusion one resolution rule (§2.2): scopes are direct perms, roles expand
    Audit gap authz_decisions_total + structured deny logs from day one

    6. Staged delivery (when scheduled)

    1. Model + service: Permission conventions, PermissionResolverProtocol + StaticPermissionResolver, AuthorizationService (is_allowed/require), AuthorizationError, metrics. Pure, unit-tested.
    2. API enforcement: requires(permission) dependency + AuthorizationService wiring at the composition root; dry-run mode.
    3. Apply to feature-platform routes + any other sensitive routes.
    4. Provider resolver (DB/Vault) + admin UX — only if a dynamic-org need arises.

    7. Decisions — all three configurable (settled)

    Per the configurability principle, none is an either/or — each is a settings / provider choice with the recommended option as the default:

    • Granularity — the permission grammar is hierarchical and supports both natively: resource:action (operation) and resource:action:instance (resource). The wildcard matcher means a principal holding features:promote (or features:*) satisfies features:promote:customer_risk. A route declares whatever granularity it needs; a require_resource_scope setting (default false) can additionally require the instance segment for chosen resources. Default routes use operation-level; resource-level is available, not mandatory.

    • Rollout — AuthorizationSettings.mode: "disabled" | "dry_run" | "enforce" (default dry_run). dry_run logs + meters authz_decisions_total{result= would_deny} without raising; enforce raises AuthorizationError (→ 403); disabled is fully permissive (loud startup warning). Switch via settings.

    • Resolver source — PermissionResolverRegistry (Strategy + Registry, like every other pluggable concern): static (default — committed TOML/settings mapping) and a db/provider resolver as a registerable seam (built when a dynamic-org need arises). Selected by AuthorizationSettings.resolver. Both supported; new resolvers register a class without touching the service (OCP).

    This makes the authz layer flexible-by-default and consistent with the rest of the framework. The build can proceed against §6's staging whenever scheduled — feature-platform routes are the first consumer.
