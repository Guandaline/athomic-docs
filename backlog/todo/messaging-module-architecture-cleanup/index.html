    Messaging module — architecture cleanup (DI, encapsulation, factory consistency)

    Status: TODO — not started · Owner: messaging/integration platform · Created: 2026-06-22

    Context

    An external architecture review of src/nala/athomic/integration/messaging/ was performed and cross-checked against the actual code. Most findings hold up; this backlog tracks only the verified, actionable ones. Two claims from the review were found inflated on inspection and are explicitly excluded (see "Not in this backlog").

    Verified findings

    1. MessageHeaders fallback bug (trivial)

    headers.py:37 does:

    for key, value in raw_headers or {}

    but the declared type (headers.py:24) is List[Tuple[str, bytes]]. Falling back to {} is semantically wrong for a list-typed field — it happens to not crash today (iterating an empty dict yields nothing, same as an empty list), but it's a latent bug waiting for someone to call .append() or otherwise treat the fallback as the declared type.

    Fix: raw_headers or [].

    2. Strategies reach into producer._publish() (encapsulation leak)

    • strategies/direct.py:102 — await producer._publish(topic=topic, message=processed_payload, ...)
    • strategies/delayed.py:153 — await producer._publish(topic=delay_topic, message=processed_envelope, ...)
    • _publish() is declared as a protected abstractmethod on BaseProducer (producers/base.py:129).

    Strategies are a collaborator of the producer, not a subclass — reaching into an underscore-prefixed method means the Strategy pattern already knows implementation details it shouldn't. This blocks swapping BaseProducer's internals without also auditing every strategy.

    Fix: promote _publish() to a public method on BaseProducer (or a narrow TransportProtocol.send() injected into strategies) that strategies are meant to call. Decide naming once during implementation, not per-call-site.

    3. MessagingManagerFactory singleton vs. ProducerManagerFactory non-singleton (inconsistency)

    • factory.py:20,38-40,58 — MessagingManagerFactory.create() caches and returns cls._instance.
    • manager_factory.py:17-19 — ProducerManagerFactory has a comment explicitly stating it no longer manages a singleton instance, and manager_factory.py:34 constructs a fresh instance every call.

    Two factories in the same module disagree on lifecycle semantics with no documented reason. This is the proximate cause of "two messaging runtimes in the same process" being effectively impossible today, independent of the broader get_settings() issue in Task 4.

    Fix: either make MessagingManagerFactory non-singleton (consistent with ProducerManagerFactory), or — if the singleton is intentional (e.g. one messaging runtime per process by design) — document that constraint explicitly in the factory's docstring and explain why ProducerManagerFactory doesn't need the same constraint. Don't silently leave the asymmetry.

    4. get_settings() called pervasively across the module (Service Locator, not DI)

    Confirmed via grep across the module — at least 14 sites including: - factory.py:46 - producers/factory.py:72 - strategies/base.py:34,39 - tasks.py:50 - retry/dlq_builder.py

    Components that look pure on the surface (e.g. ProducerFactory.create()) implicitly depend on global settings for serializer, lineage, and causality config instead of receiving them via constructor injection. This is the structural blocker for: - two independently configured messaging runtimes in one process (e.g. different serializers per tenant), - testing strategies/factories without monkeypatching get_settings(), - per-tenant messaging config (relevant given the project's existing multi-tenancy work in ai/features).

    Fix (larger, do last): thread serializer, lineage_settings, and causality_settings (or a single MessagingContext bundling them) through constructors of ProducerFactory, BasePublishingStrategy, and the republish task path, instead of each component calling get_settings() independently. This is a multi-file, higher-effort change — sequence it after Tasks 1–3 land, since those are small and unblock incrementally.

    5. generic_republish_message_task builds a fresh ProducerManager per invocation

    tasks.py:62 — producer_manager = ProducerManager(settings=connection_settings) constructed inline in the task body, with no cache/pool. Every retry re-builds the serializer, strategies, and producer connection from scratch. Depending on backend (e.g. Kafka producer + compression pipeline + tracing adapters), this is real per-retry overhead, not just an aesthetic concern.

    Fix: introduce a small keyed cache/pool (e.g. RepublishProducerRegistry or similar, keyed by connection name) so retries reuse an existing ProducerManager instead of reconstructing one. Needs a lifecycle story (when entries are evicted/closed) — don't just cache forever.

    6. Startup/shutdown asymmetry in MessagingManager

    • manager.py:142-143 — startup is sequential: await service.connect() then await service.wait_ready() per subordinate, one at a time.
    • manager.py:153-156 — shutdown is parallel via asyncio.gather(*(s.stop() for s in reversed(...))).

    Not necessarily wrong (startup may have real ordering dependencies shutdown doesn't), but it's undocumented — there's no comment explaining why the two phases use different concurrency models. Worth a deliberate decision: either the dependency order matters and should be explicit (so the next person doesn't parallelize startup and break something silently), or it doesn't and startup should be parallelized too for faster boot.

    Fix: investigate actual subordinate dependencies (does DLQManager need ProducerManager connected first? does RepublisherManager?) and either (a) parallelize startup where there's no real dependency, or (b) add a short comment at manager.py:142 stating the ordering constraint explicitly.

    7. Causality (VectorClock) instantiated directly inside the publishing strategy

    strategies/base.py:85-87 — BasePublishingStrategy._inject_causality_headers() instantiates VectorClock() directly and calls current_vc.increment(app_name) inline. There's no seam to swap in a LamportClock or HybridLogicalClock without editing the strategy itself.

    Fix: extract a CausalityProvider protocol (next() -> CausalityToken or similar) and inject it into BasePublishingStrategy, with VectorClock as the default registered implementation — consistent with this project's existing Strategy+Registry convention (see CLAUDE.md design pattern table). Lower priority than Tasks 1-3 since it's not blocking anything today, just inflexible.

    Suggested order

    1. Task 1 (MessageHeaders fallback) — trivial, do first.
    2. Task 2 (_publish() encapsulation) — small, isolated.
    3. Task 3 (factory singleton inconsistency) — decide + document or fix.
    4. Task 6 (startup/shutdown asymmetry) — investigation + either fix or comment.
    5. Task 5 (republish producer pooling) — moderate, needs lifecycle design.
    6. Task 7 (CausalityProvider extraction) — moderate, no urgency.
    7. Task 4 (get_settings() → DI) — largest, do last; Tasks 1-3 reduce noise before touching every constructor in the module.

    Acceptance

    • make test and make mypy green after each task.
    • No behavioral change for Tasks 1-3, 6, 7 (pure refactors/fixes).
    • Task 4 and 5 need new unit tests covering: two independently configured producers/strategies in the same process (Task 4), and producer reuse across retries (Task 5).

    Not in this backlog

    The following claims from the original review were checked against the code and found overstated — not tracked here:

    • DLQHandler as a single monolithic class mixing 5 responsibilities. In reality it's ~276 lines with work already delegated to a separate retry/dlq_builder.py factory for envelope construction. It's a candidate for future SRP cleanup but isn't the structural risk the review implied — re-evaluate only if the file keeps growing.
    • BaseProducer as an 8-responsibility god class. It has several injected collaborators (telemetry, topic resolver, strategy selection) but publish() itself just delegates to the selected strategy (producers/base.py:101-102,119) — the Strategy pattern already absorbs most of the complexity the review attributed to this class. No split planned unless new responsibilities get added directly to publish() itself rather than via a new strategy/collaborator.

    Also not in scope: the review's larger restructuring proposals ("MessagingRuntime vs MessagingFacade" split, full ServiceGraph DAG scheduler for startup) — those are bigger architectural bets, not inconsistencies or bugs, and should go through a separate design discussion if pursued.
